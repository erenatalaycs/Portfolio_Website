# Phase 3 — UI Design Contract

> Visual and interaction contract for Phase 3 (Content Integration + MDX Write-ups + Camera Choreography). Generated by gsd-ui-researcher, verified by gsd-ui-checker. **This document EXTENDS the Phase 1 + Phase 2 UI-SPECs; it does NOT redefine the palette, typography scale, spacing scale, `<BracketLink>` contract, OrbitControls clamps, or any existing component contract.** Every locked decision in `03-CONTEXT.md` (D-01..D-20) is honoured here, not re-debated. The 6-pillar checker will verify Phase 1 + Phase 2 carryover.

## Inheritance Notice

The Phase 1 + Phase 2 UI-SPECs are the canonical source of truth for:

- **Palette tokens** — eight-color `@theme` block (`--color-bg #0d1117`, `--color-fg #c9d1d9`, `--color-accent #7ee787`, `--color-warn #e3b341`, `--color-negative #ff7b72`, `--color-muted #8b949e`, `--color-surface-1 #161b22`, `--color-focus #79c0ff`).
- **Typography scale** — JetBrains Mono, 4 sizes (14/16/20/24px), 2 weights (400/600), body line-height 1.5, body max-width 72ch.
- **Spacing scale** — Tailwind v4 default 4-multiple scale; 44×44 touch-target rule on every interactive element.
- **`<BracketLink>` contract** — five states (default / hover / focus-visible / active-click / `active` prop), `as: 'a' | 'button'` discriminated union, `aria-pressed` on toggle buttons.
- **3D scene palette mapping** — `src/scene/colors.ts` mirror of CSS tokens (`bg`, `surface`, `accent`, `warn`).
- **OrbitControls clamps** — `'orbit'` mode `minPolarAngle=π/3`, `maxPolarAngle=100°`, `minAzimuthAngle=-π/2`, `maxAzimuthAngle=+π/2`, `minDistance=1.2`, `maxDistance=4.0`, `enablePan=false`, `enableDamping=true`, `dampingFactor=0.08`, `target=[0, 0.6, 0]`.
- **Default 3D camera** — `position=[2.4, 1.4, 2.4]`, `fov=50`, looking at `[0, 0.6, 0]`.
- **Procedural workstation primitives** — desk + 3 monitors at `±0.45/0` x-spacing, `y=0.95`, `z=-0.05` with ±0.18 rad inward angle on outer monitors; lamp at `[-0.5, 0.78, 0]`; bookshelf at `[0, 0.75, -0.6]`.
- **Anti-pattern audit** — Matrix rain, padlock/shield motifs, hooded-hacker silhouettes, skill bars, type-out on body content, `#000000`+`#00ff41`, fake REPL, in-canvas text via drei `<Text>`, walls/ceiling/sky, textures on procedural primitives, custom GLSL shaders, permanent Glitch/DOF/SSAO postprocessing — ALL still forbidden in Phase 3.

Phase 3 ADDS: (a) drei `<Html transform occlude="blending">` overlays on each monitor's screen plane, (b) animated `whoami` greeting on center monitor, (c) GSAP click-to-focus camera animation per monitor with Esc/outside-click/re-click defocus, (d) `?focus=<id>` URL deep-link extension for the 3D shell, (e) MDX pipeline with three new MDX renderer components (`<CodeBlock>`, `<ScreenshotFrame>`, `<ProvenanceFooter>`), (f) right-monitor write-ups list with in-place write-up rendering and back button, (g) ~10 new copy strings for write-up index headings, back-to-list button, and provenance footer labels. Nothing else changes. Postprocessing (Bloom + Scanline + ChromaticAberration + Noise) remains forbidden in Phase 3 — that's Phase 4 (3D-08).

## Spacing Scale

**Inherited verbatim from Phase 1 + 2.** No new spacing tokens introduced by Phase 3.

The new components Phase 3 ships use only existing 4-multiple tokens:

| Component / Site | Spacing usage |
|------------------|---------------|
| `<MonitorOverlay>` (the drei `<Html>` content frame) | Internal padding `p-4` (16px) — comfortable margin from the 3D-rendered bezel without wasting screen real estate. NO outer margin on the `<Html>` itself (drei positions it precisely on the 0.55×0.32m screen plane). |
| `<WhoamiGreeting>` line spacing | `leading-relaxed` (1.5) inherited from body type; inter-line whitespace between `$ whoami` / `$ status` / `$ links --all` blocks is a literal `\n` blank line in the rendered `<pre>`, NOT a margin (terminal output spacing behaviour). |
| `<MonitorScroll>` (scrollable container inside center / right monitor) | `space-y-4` (16px) between top-fixed `<WhoamiGreeting>` block and scrollable About / Skills sections. Scroll thumb: native `overflow-y-auto`; no custom scrollbar styling in Phase 3 (Phase 4 may revisit). |
| MDX prose body | `space-y-4` (16px) between paragraphs. `space-y-6` (24px) between subsections (e.g. between "Hypothesis" and "Telemetry" headings). Same rhythm as Phase 1's About / Education paragraphs — recruiter reads the text shell and an MDX write-up with congruent vertical rhythm. |
| MDX `<h2>` (write-up section heading) → first body element | `mt-6` (24px above) `mb-3` (12px below) — same 12px tightening exception documented in Phase 1 between section heading and its first paragraph (the `<h2>` and its body read as one stanza). |
| `<CodeBlock>` outer wrapper | `my-4` (16px above + below) — sits inline in MDX prose flow with one body-paragraph gap. Internal padding `px-4 py-3` (16px / 12px) inside the code area; filename caption bar `px-4 py-2` (16px / 8px). |
| `<ScreenshotFrame>` outer wrapper | `my-4` (16px above + below). Internal padding `p-2` (8px) around the `<img>` so the image doesn't touch the bordered frame. Caption: `mt-2` (8px) below the image. |
| `<ProvenanceFooter>` separator | `mt-12` (48px) above the footer (a clear inter-section gap signalling "this is the metadata"); internal `space-y-2` (8px) between the three footer blocks (sources / ATT&CK / author+date). |
| Write-up list rows (right monitor `<WriteupList>`) | `gap-y-3` (12px) between rows — same 12px tightening exception as Phase 1's heading-to-body. Tight enough to scan a 5-10 entry list at-a-glance, loose enough that rows don't visually merge. |
| `[back to list]` button | Anchored at `top-4 left-4` (16px / 16px) inside the right-monitor `<MonitorOverlay>` when in write-up view mode — does NOT scroll with content, stays accessible as the user reads. |

**Exceptions to the existing exception list:** none added. The 12px tightenings above (`mb-3` after `<h2>`, `gap-y-3` between list rows) reuse the single Phase 1 exception (`mt-3` between section heading and first paragraph), not a new exception.

**44×44 touch-target rule applies to:** every interactive element rendered inside drei `<Html>` overlays — `[CV]` `[GitHub]` `[LinkedIn]` `[email]` brackets in `<WhoamiGreeting>`, every write-up row in `<WriteupList>`, `[back to list]` button, `[copy]` button on `<CodeBlock>`, every link inside MDX prose. drei `<Html transform>` does not interfere with hit-testing — DOM events fire normally, so the existing `py-2 px-3 -mx-1` `<BracketLink>` styling carries over verbatim.

## Typography

**Inherited verbatim from Phase 1 + 2.** Body 16px / 400 / 1.5, Label 14px / 400 / 1.4, Heading 20px / 600 / 1.3, Display 24px / 600 / 1.3.

Phase 3 introduces **zero new type roles**. Specifically:

| Phase 3 element | Type role used | Rationale |
|-----------------|----------------|-----------|

Body 16px keeps it readable at the calibrated `distanceFactor` (drei docs: 14-16px is the readable floor at 0.65-0.70m default-camera distance) and matches "real terminal output" rhythm. The text-shell `<Hero>` keeps its 20px/600 — they are different render contexts. |
| `<WhoamiGreeting>` output lines (the `Eren Atalay — Junior Cybersecurity Analyst (UK)` etc.) | **Body 16px / 400 / 1.5** | Same as prompt lines for visual congruence. |
| `[CV]` `[GitHub]` `[LinkedIn]` `[email]` brackets inside `<WhoamiGreeting>` | **Body 16px / 400 / 1.5** | Inherits `<BracketLink>` Phase 1 styling. |
| Center-monitor scrollable About paragraphs | **Body 16px / 400 / 1.5** | Same as text shell. Reuses `<About />` component verbatim — single source of truth (D-01). |
| Center-monitor Skills tag list | **Label 14px / 400 / 1.4** | Reuses `<Skills />` component → `<SkillTag>` 14px verbatim. |
| Right-monitor `<WriteupList>` row title | **Body 16px / 400 / 1.5** | The write-up title is the click target — body type for legibility. |
| Right-monitor `<WriteupList>` row metadata (date — duration — type-marker) | **Label 14px / 400 / 1.4** | Same role as Phase 1 cert dates / project repo URLs. Renders in muted color (`#8b949e`). |
| MDX `<h1>` (write-up title) | **Heading 20px / 600 / 1.3** | Matches Phase 1 section heading scale. NO `<h1>` jumps up to Display 24px — write-ups are not 404-loud. |
| MDX `<h2>` (write-up section heading: Hypothesis / Telemetry / Sigma Rule / etc.) | **Heading 20px / 600 / 1.3** | Same as `<h1>`; visual hierarchy comes from the prompt-prefix styling (`$ ` accent green leader on each heading), not type-size step. |
| MDX `<h3>` (rare — only used inside long sections) | **Body 16px / 600 / 1.5** | Heavier than body, smaller than `<h2>`. Optional in Phase 3; if a write-up doesn't need `<h3>`, don't introduce it. |
| MDX prose `<p>` | **Body 16px / 400 / 1.5** | Same as text shell. Max line length 72ch via `<MDXRenderer>` outer wrapper. |
| MDX `<code>` inline (single-token like `certutil.exe`) | **Body 16px / 400 / 1.5** | Same as prose; inherits monospace font from the document. Visual differentiation comes from a subtle `bg-surface-1` (`#161b22`) with `px-1` (4px horizontal padding) — the SAME alt-surface used for the Phase 2 ContextLossBar and Phase 1 row hover state. NO accent or warn color on inline code. |
| `<CodeBlock>` body code | **Label 14px / 400 / 1.4** (slight step-down from prose 16px) | Code blocks are dense; 14px monospace at 1.4 line-height is the recruiter-readable density for 60-80 char lines without horizontal scroll on a 600×400 monitor render. Outside the monitor (full-page write-up route in text shell), the same 14px holds — congruent rhythm. Shiki tokenises at this size; no responsive step. |
| `<CodeBlock>` filename caption | **Label 14px / 400 / 1.4** | Smaller than the code body would feel inverted; same size keeps it visually balanced as a chrome strip on top of the code. Renders in muted color (`#8b949e`). |
| `<CodeBlock>` language badge | **Label 14px / 400 / 1.4** | Same row as filename caption; right-aligned. Renders in muted color. NO accent — language is metadata, not voice. |
| `<CodeBlock>` `[copy]` button | **Label 14px / 400 / 1.4** | Same row as caption + badge; `<BracketLink as="button">` styling — accent green like every other bracket. |
| `<ScreenshotFrame>` caption | **Label 14px / 400 / 1.4** | Below the image. Renders in muted color. |
| `<ScreenshotFrame>` `[sanitized]` badge | **Label 14px / 400 / 1.4** | Inline at end of caption. Renders in **accent** color (`#7ee787`) — this is voice, asserting OPSEC compliance. Wrapped in `[brackets]` for consistency. **This is a NEW accent use — see Color section.** |
| `<ProvenanceFooter>` heading (`# sources` / `# attack-techniques` / `# meta`) | **Body 16px / 400 / 1.5** | Comment-prefix style (literal `# ` muted-color leader, then the section name in body color) — same pattern as Phase 1's `# No write-ups published yet` empty-state comment. NOT Heading 20px because it's footer-metadata, not a content heading. |
| `<ProvenanceFooter>` body (source list, ATT&CK ids, author+date) | **Label 14px / 400 / 1.4** | Footer metadata stays denser than prose — recruiter scans, doesn't read line by line. |

**No type-out animation on body content** (forbidden in Phase 1, reaffirmed for Phase 3). The single Phase 3 typing animation is the `<WhoamiGreeting>` sequence on the center monitor — sequenced character-by-character across the THREE prompt lines only (`$ whoami`, `$ status`, `$ links --all`); the OUTPUT lines below each prompt appear instantly as a real terminal would render them. After the sequence completes, the cursor `█` blinks at 1Hz and the sequence does NOT loop (D-07 — anti-pattern: confuses recruiters who arrive mid-loop).

**Body max-width / line length:** unchanged at **72ch**. Inside the `<MonitorOverlay>` overlays, the 600px DOM width naturally clamps to ~80 characters of monospace at 14-16px — recruiter reads inside the monitor at the same comfortable density as the text shell. For full-page write-up routes (text shell `?focus=writeups/<slug>` rendering, if planner adds one), the 72ch wrapper from Phase 1's `<main>` applies.

The 3D scene palette mirror (`src/scene/colors.ts`) gains zero new entries — Phase 3 ships no new in-canvas colored geometry; the existing emissive accent on monitor screens (Phase 2 D-06) stays exactly as it is. drei `<Html transform>` overlays render DOM with full CSS palette access, so all text inside monitors uses the existing CSS custom properties.

### Accent reserved for (Phase 3 additions to Phase 1 + 2's list)

In Phase 1, accent (`#7ee787`) was reserved for 8 specific text uses. Phase 2 added 3 (toggle-active inversion, retry link, in-canvas monitor screen emissive). Phase 3 extends the list with **six new in-DOM uses** — every single one is on this list:

14. **`<WhoamiGreeting>` `[CV] [GitHub] [LinkedIn] [email]` brackets** — already on the Phase 1 list (item 3) for the text-shell hero; the in-monitor render is the same component (`<Hero />` reused or `<WhoamiGreeting>` wrapping the same `<BracketLink>` + `<EmailReveal>` pieces) → inherits the accent treatment without a new entry. Listed here for redundant clarity.
15. **MDX `<h1>` and `<h2>` prompt prefix** — the `$ ` leader on every write-up section heading (e.g., `$ cat hypothesis.md`, `$ cat telemetry.md`). Identical to the Phase 1 `<TerminalPrompt>` accent treatment for `$ cat about.md` headings; reuses the component verbatim.
16. **MDX inline links** (text-shell write-up routes + in-monitor write-up renders) — `<BracketLink>` style: accent green bracket-and-label, hover underline, focus ring blue. NO color change between text-shell rendering and in-monitor rendering — single source of truth.
17. **`<ScreenshotFrame>` `[sanitized]` badge** — inline accent-green bracket at the end of the screenshot caption, asserting "this image was reviewed against the OPSEC checklist before commit." This is **voice** — the cybersecurity portfolio's commitment to OPSEC discipline made visible. The check-mark glyph used inside the brackets is the literal character `✓` (U+2713), the SAME glyph used in Phase 1 for the `[✓]` earned-cert leading bracket. Pattern: `[✓ sanitized]`.
18. **`<CodeBlock>` `[copy]` button + post-copy `[copied]` state** — `<BracketLink as="button">` styling, accent green. Hover underline, focus ring blue. On successful clipboard write, button label briefly changes to `[copied]` for 1.5s (same pattern as Phase 1's hero email-reveal `(copied)` toast), then restores to `[copy]`. NO color change for the post-copy state — same accent green; the label itself communicates the state change.
19. **`<WriteupList>` row title link** — each write-up row's title text wrapped in `<BracketLink>` → accent green, hover underline. Type-marker glyph (`[D]` / `[I]` / `[W]`) leading the row also renders in accent green, matching the bracket-glyph language. Date and duration metadata renders in muted color, NOT accent.
20. **`<WriteupView>` `[back to list]` button** — `<BracketLink as="button">` accent green styling. Anchored to the top-left of the in-monitor write-up view; clicking returns the right-monitor render to `<WriteupList>` with no camera animation (the user is already focused on the right monitor; the in-monitor content swaps).

### Warn reserved for (Phase 3 additions to Phase 1 + 2's list)

In Phase 1, warn (`#e3b341`) was reserved for 1 use (cert "in progress" leading bracket). Phase 2 added 2 (`[!]` glyph in ContextLossBar, in-canvas lamp bulb emissive). Phase 3 extends with **zero new uses**. Reasoning: write-ups aren't "in progress" or "warning" — they're either published (rendered) or not yet authored (the empty-state copy uses muted color, not warn, since "no write-ups yet" is informational, not cautionary). The Provenance footer's "Home Lab — Synthetic Data" disclaimer (when applicable) renders in **muted** color, NOT warn — it's metadata, not a warning.

### Negative reserved for (Phase 3 additions)

In Phase 1, negative (`#ff7b72`) was reserved for 1 use (the 404 page's `bash: cd: ... No such file or directory` line). Phase 2 added zero. Phase 3 extends with **zero new uses**. Specifically: the in-monitor empty state (e.g., recruiter lands at `?focus=projects` but `projects.ts` is empty) does NOT use negative — it uses the same `# No projects published yet — see [GitHub] for ongoing work.` copy as Phase 1, in muted color. A missing write-up (deep-link to a slug that doesn't exist) re-uses the Phase 1 404 page — this is the only Phase 3 negative-color rendering, inherited not new.

### NOT permitted in Phase 3 (anti-uses to police in review)

- **Accent on MDX prose body text** — body uses `#c9d1d9`, never accent green. Inline `<code>` uses body color on `bg-surface-1` (`#161b22`), never accent.
- **Accent on the `<CodeBlock>` filename caption / language badge** — both render in muted color (`#8b949e`). Only the `[copy]` button uses accent. Reasoning: filename and language are metadata, not voice; making them accent would dilute the "accent reserved for voice" rule.
- **Accent on `<WriteupList>` row metadata** — date (`2026-05-15`), duration (`~12 min read`) render in muted color. Only the title and the type-marker use accent.
- **Accent on Shiki syntax-highlighted tokens that aren't naturally green in `github-dark`** — the `github-dark` theme tokens render at their theme-defined colors (typically green for strings, orange for numbers, red for keywords, blue-ish for functions). These are **already** part of the Shiki theme and are scoped to `<CodeBlock>` rendering only. They do NOT count as palette additions because: (a) they're build-time-baked into the HTML by `rehype-pretty-code`, (b) they only appear inside `<CodeBlock>` elements, never in surrounding prose, (c) `github-dark` is a known recruiter-familiar theme. Shiki's accent-token green is INTENTIONALLY similar to but not identical to `--color-accent` — that's fine, they live in different contexts.
- **Negative on any error / warn state in Phase 3** — see above. The `<ContextLossBar>` from Phase 2 is the only orange `[!]` site-wide; Phase 3 introduces no new error UI surface.
- **Backlit "screen glow" tinting MDX prose** — write-up content inside drei `<Html transform>` is plain DOM, NOT glow-tinted by the underlying emissive screen plane. The emissive accent green from the Phase 2 `<Monitor>` plane is occluded by the opaque `<Html>` overlay (occlude="blending" handles depth correctly so the DOM hides the backlight where the DOM is opaque, and the backlight bleeds through where the DOM has transparent regions — there should be NO transparent regions in Phase 3 monitors; `<MonitorOverlay>` renders a fully opaque `bg-bg` surface).
- **`bg-accent` (solid accent) on any block-level container** — accent stays a *text/inline* color in Phase 3 except where Phase 1 + 2 already authorised it (the `:active` flash on `<BracketLink>`, the persistent `active` prop inversion on toggle buttons, the in-canvas monitor emissive). No block-level accent fills introduced in Phase 3.

## Copywriting Contract

### `<WhoamiGreeting>` — center monitor (D-05, D-06, D-07)

The exact rendered text inside the center monitor's drei `<Html transform>` overlay. Maps to Phase 1's hero block, expanded to three prompt lines per D-05. The `[CV]` `[GitHub]` `[LinkedIn]` `[email]` brackets are the SAME real `<a>` and `<button>` elements as Phase 1 — same `<BracketLink>` and `<EmailReveal>` components, single source of truth (D-01).

```
$ whoami
eren@workstation:~$
Eren Atalay — Junior Cybersecurity Analyst (UK)

$ status
Currently: QA at Fin-On — studying Security+ — available for SOC analyst roles

$ links --all
[CV]  [GitHub]  [LinkedIn]  [email (click to reveal)]
█

### Camera-focus visual indicators (D-08, D-10)

**Locked: NO on-screen "press Esc to defocus" hint, NO focus-state DOM overlay.** Cinematic without HUD chrome.

Reasoning:
- The 3D shell is already exploratory — the user clicked a monitor on purpose; they're paying attention. A persistent on-screen hint is patronising.
- An overlay would compete with the in-monitor `<Html>` content for visual attention — exactly the wrong attention shift after the user just decided to read what's on a monitor.
- Defocus triggers (Esc / outside-click / re-click) are intuitive enough that recruiters discover them naturally. (Outside-click especially — clicking empty space on a Canvas conventionally backs out of focus.)
- Reduced-motion users get instant cuts, so they can't be confused by an unfinished animation.

**Hidden interaction surface for outside-click defocus** (D-10 trigger #2):
- The Canvas itself acts as the defocus surface — clicking ANY pointer event on the Canvas that doesn't hit a monitor screen plane (raycaster check) triggers defocus. NO visible "click here to escape" overlay element.
- Recommendation for planner: use a raycaster check on click events. If `raycaster.intersectObjects([screenPlane1, screenPlane2, screenPlane3])` returns empty → defocus. Otherwise → focus the hit monitor (or re-click defocus if already focused).

**Esc keyboard hint copy: NONE on first focus.** A recruiter pressing Esc once to dismiss is the universal browser pattern; we don't restate the obvious. If user testing surfaces confusion, a single muted-color hint line `# press esc to return` could be added at the BOTTOM of the focused-monitor's `<Html>` content (NOT a Canvas-overlay) — defer this decision to Phase 4 polish.

### `?focus=<id>` URL deep-link arrival behaviour (D-11)

When a recruiter lands directly at `?view=3d&focus=projects` (e.g., from a CV link or shared URL):

- **Locked: instant snap to focus pose, NO cinematic intro animation.** Same animation contract as a click triggered from inside the scene — `1000 ms power2.inOut` for click triggers — BUT for **first-load deep-link**, the Phase 2 "no camera intro animation on first 3D mount" rule applies: the camera renders directly at the focused monitor's preset pose with NO animation.

## Layout & Interaction Contract

### `<Monitor>` refactor for `<Html transform>` mount (D-01, D-03)

Phase 2's `<Monitor>` ships the screen plane as a fixed-position child mesh. Phase 3 needs the screen plane to be a **named ref or accept `children`** so the `<Html transform>` can mount on the correct plane. Two options for the planner:

**Option A (recommended):** `<Monitor>` accepts a `children` prop. The screen plane mesh stays inside `<Monitor>`; children render *after* the screen plane mesh as a sibling at the same `position={[0, 0, 0.025]}` (slightly in front of the plane to avoid z-fighting). `<Workstation>` passes the relevant `<Html transform>` as a child:

```tsx
<Monitor position={[-0.45, 0.95, -0.05]} rotation={[0, 0.18, 0]}>
  <Html transform occlude="blending" position={[0, 0, 0.026]} ...>
    <ProjectsOverlay />
  </Html>
</Monitor>
```

**Option B:** `<Monitor>` exposes a `screenRef: RefObject<Mesh>` via forwardRef; the `<Html>` mounts at `<Workstation>` level using the ref. More indirection; less idiomatic R3F. Not recommended.

**Locked: Option A.** The `<Monitor>` Phase 2 contract is a thin extension — adds an optional `children?: ReactNode` prop, no breaking change to the existing `<Workstation>` call sites.

### `<MonitorOverlay>` — drei `<Html transform>` wrapper contract (D-02, D-03)

The single component that wraps every monitor's DOM content. Owns:
- The `<Html transform occlude="blending">` props.
- The 600×400 px outer frame size and the calibrated `distanceFactor`.
- The opaque dark-bg backplate (so the emissive accent screen behind doesn't bleed through).
- The internal scrollable region.

```tsx
// src/scene/MonitorOverlay.tsx (Phase 3 ships)
import { Html } from '@react-three/drei';

interface MonitorOverlayProps {
  children: ReactNode;
  ariaLabel: string; // for the wrapped <div>'s a11y name
}

export function MonitorOverlay({ children, ariaLabel }: MonitorOverlayProps) {
  return (
    <Html
      transform
      occlude="blending"
      position={[0, 0, 0.026]} // 1mm in front of screen plane to avoid z-fighting
      distanceFactor={DISTANCE_FACTOR}
      style={{ width: '600px', height: '400px' }}
      wrapperClass="monitor-overlay-wrapper"
    >
      <div
        role="region"
        aria-label={ariaLabel}
        className="w-full h-full bg-bg text-fg font-mono p-4 overflow-y-auto"
      >
        {children}
      </div>
    </Html>
  );
}
```

| Prop / style | Value | Rationale |
|--------------|-------|-----------|
| `<Html>` `transform` | `true` | Projects DOM onto a 3D plane via CSS `matrix3d`. Required for occlusion correctness (D-03 + Phase 2 D-06 emissive backing). |
| `<Html>` `occlude` | `"blending"` | Per CLAUDE.md drei pinning — the depth-correct occlusion mode. Monitor frame correctly hides DOM when camera orbits behind. |
| `<Html>` `position` | `[0, 0, 0.026]` | 1mm in front of the screen plane (`z=0.025` per Phase 2 Monitor.tsx). Z-fighting prevention. |
| `<Html>` `distanceFactor` | calibrated by planner — research-confirmed using drei docs; principle: 14px text readable from `[2.4, 1.4, 2.4]` default camera position (~3.4m away from monitor at `[0, 0.95, -0.05]`). Estimate `~1.5-2.5`; planner verifies in dev. | Per D-03 — body text legibility from default orbit pose. |
| inner `<div>` width × height | `600px` × `400px` | Per D-03 real-pixel-density. |
| inner `<div>` `bg-bg` | `--color-bg` (`#0d1117`) | Opaque backplate. Hides the Phase 2 emissive-accent screen behind. **No transparent regions** in the overlay (would let the emissive backlight bleed through and tint MDX prose — anti-pattern). |
| inner `<div>` `font-mono` | inherited JetBrains Mono | DOM inside `<Html>` retains CSS — same font family as the text shell. |
| inner `<div>` padding `p-4` | 16px | Comfortable margin from the monitor's bezel edge. |
| inner `<div>` `overflow-y-auto` | scrollable | Per D-02 — mouse wheel scrolls the focused monitor's DOM, not the 3D scene. |
| inner `<div>` `role="region"` `aria-label={ariaLabel}` | required | Screen-reader names the monitor's content (e.g., `aria-label="Center monitor: identity and skills"`, `aria-label="Right monitor: write-ups"`). |

**Forbidden in MonitorOverlay:**
- `bg-transparent` or any opacity < 1.0 — emissive bleed-through is the anti-use.
- Any `border` or `outline` on the inner `<div>` — the 3D Monitor frame primitive (Phase 2) provides the bezel; a DOM border would create a doubled frame.
- Custom scrollbar styling — native scrollbar only in Phase 3; Phase 4 polish may revisit.
- Any `<iframe>` — drei `<Html>` documented to have iframe interaction blind spots at oblique angles (CLAUDE.md decision rationale). Not relevant in Phase 3 — write-ups render via MDX as React components, not as embedded iframes.

### `<FocusController>` — camera focus state machine (D-08, D-09, D-10)

A new component (or zustand store, planner's call — recommendation: zustand for cross-component access from `<OrbitControls>`, `<Canvas>` click handlers, header-link click handlers, AND `<MonitorOverlay>` re-click handler):

**State:**
- `focused: 'left' | 'center' | 'right' | null` — null is default orbit pose; otherwise indexes into `MONITOR_FOCUS_POSES`.
- `isAnimating: boolean` — true during the 1000ms GSAP timeline; gates click handlers from re-firing.

**Per-monitor focus poses (D-08):**

| Monitor | Camera position | LookAt / OrbitControls target |
|---------|-----------------|-------------------------------|
| Left | `[-0.45, 1.0, 0.7]` | `[-0.45, 0.95, 0]` (the screen plane center) |
| Center | `[0, 1.05, 0.65]` | `[0, 0.95, -0.05]` |
| Right | `[0.45, 1.0, 0.7]` | `[0.45, 0.95, 0]` |
| Default (orbit) | `[2.4, 1.4, 2.4]` | `[0, 0.6, 0]` |

Planner verifies these positions place the focused monitor at ~70% of the viewport vertical via the `distanceFactor` from D-03. If the calibrated `distanceFactor` produces 14px text legibility at distance >0.7m, planner adjusts the focus-pose `position` z-component accordingly.

### Click-to-focus animation (D-08)

Triggered by:
1. **Pointer click on a monitor's screen plane** — raycaster hit on `screenPlane1 | screenPlane2 | screenPlane3` → `setFocused('left' | 'center' | 'right')`.
2. **`<BracketLink>` in header `$ goto […]` while in 3D shell** — clicking `[projects]` → `setQueryParams({ focus: 'projects' })` → `<FocusController>` listens to `qpchange` → `setFocused('left')`. The header-link interaction is the SAME code path as the in-canvas click; the animation is identical.
3. **`?focus=<id>` URL on first 3D mount** — the controller reads `useQueryParams()` on mount; if `focus` matches a monitor's id, **snaps directly** to that monitor's pose (NO animation — see D-11 + the `?focus=<id>` URL deep-link arrival section above).

**Animation properties:**
- **Duration:** 1000 ms.
- **Easing:** `power2.inOut`.
- **Animated:** `camera.position` + `OrbitControls.target` simultaneously, single GSAP timeline (planner's call vs two parallel timelines — recommendation: single timeline for atomic "focused/not-focused" state).
- **OrbitControls during animation:** `controls.enabled = false`. Re-enabled AFTER animation completes on defocus (D-09 + Claude's-discretion default).
- **`controls.update()` called per-frame during animation** — required so the new target position is applied each frame.
- **Reduced-motion:** `useReducedMotion() === true` → instant snap (no GSAP timeline; direct `camera.position.copy()` + `controls.target.copy()` + `controls.update()`).

### Defocus animation (D-10)

Triggered by:
1. **Esc key** — `window.addEventListener('keydown', ...)` registered on `<ThreeDShell>` mount, removed on unmount. `e.key === 'Escape' && focused !== null` → `setFocused(null)`.
2. **Click outside any monitor's screen plane** — raycaster check on canvas click; if no hit on a screen plane AND `focused !== null` → `setFocused(null)`.
3. **Re-click on the currently-focused monitor's screen plane** — if `screenPlane.userData.id === focused` → `setFocused(null)`.

**Animation properties:** identical to focus animation — 1000 ms `power2.inOut`, single timeline animating `camera.position` + `OrbitControls.target` back to their default values (`[2.4, 1.4, 2.4]` + `[0, 0.6, 0]`). OrbitControls re-enabled AFTER timeline completes (Claude's-discretion: `controls.enabled = true` in `onComplete` callback).

**URL synchronisation on defocus:**
- Defocus also clears `?focus=` from the URL via `setQueryParams({ focus: undefined })`. Single source of truth: URL ↔ focus state.
- Pressing browser back from a focused state navigates to the prior URL (which may or may not contain `?focus=`); `<FocusController>` reads the new URL on `qpchange` and animates accordingly.

### `<WriteupView>` in-place rendering (D-19)

When the right monitor is in `<WriteupView>` mode (URL `?focus=writeups/<slug>`), the focus state remains `'right'` — the camera does NOT re-animate. Only the in-monitor DOM swaps from `<WriteupList>` to `<WriteupView>`.

**Click-to-defocus while in `<WriteupView>`:**
- Pressing Esc / re-clicking the right monitor / clicking outside → SAME defocus behaviour, sets `focused = null` AND clears `?focus=` from URL (drops both the `writeups` and `<slug>` segments).
- Camera animates back to default orbit; right monitor re-renders the `<WriteupList>` because `?focus=writeups/<slug>` is gone.
- This is intentional: defocusing also "closes" the write-up view. Recruiter who pressed Esc to leave a focused monitor lands at the desk overview, sees three monitors with their default content (list, not detail).

**Re-clicking inside the focused right monitor while reading a write-up:**
- The `<MonitorOverlay>` inner `<div>` does NOT propagate clicks to the canvas (`onClick={e => e.stopPropagation()}` on the inner div) — recruiter clicking inside the write-up text to select / copy text MUST NOT trigger defocus.
- The re-click defocus trigger only fires on raycaster hits to the screen plane MESH (not to the DOM overlay above it). Z-fighting prevention (`position={[0, 0, 0.026]}` on the `<Html>`) ensures the DOM overlay is in front of the screen plane, so screen-plane raycaster hits only happen via clicks on the bezel area or empty Canvas — which is the correct re-click defocus behaviour.

### Touch / pointer behaviour (Phase 3 additions)

Phase 2 confirmed: single-finger drag = orbit, two-finger pinch = zoom, `touch-action: pan-y` on Canvas allows page scroll on touch devices.

Phase 3 ADDS:

- **Tap on a monitor screen plane** = click-to-focus. Same code path as desktop click. drei's pointer events fire `onClick` for both mouse and touch; planner uses the unified pointer event API.
- **Drag inside a focused monitor's `<Html>` overlay** = scroll the DOM (overflow-y-auto). Does NOT propagate to canvas drag. (`onPointerDown={e => e.stopPropagation()}` on the inner div.)
- **Pinch inside a focused monitor's `<Html>` overlay** = browser pinch-zoom on the projected DOM, NOT a 3D camera zoom. drei `<Html transform>` projects DOM via CSS; the browser's pinch gesture acts on the underlying CSS transform. Planner verifies on iOS Safari + Android Chrome — if pinch causes layout breakage, gate with `touch-action: pan-y` on the inner `<div>` to disable pinch (recruiter doesn't NEED to pinch-zoom inside a 600×400 monitor; the 14px text is already legible).
- **Esc key on touch devices** — touch devices don't have a physical Esc. Defocus on touch devices triggers via outside-click (tap outside any monitor) or re-tap on the focused monitor. Both are intuitive.

### Focus management & keyboard navigation (Phase 3 additions)

Phase 2's keyboard contract is preserved. Phase 3 ADDS:

1. **Esc key listener** — registered at `<ThreeDShell>` level (only relevant when 3D shell is mounted; planner's call vs App level — D-19 Claude's-discretion locked at ThreeDShell). Triggers defocus via `<FocusController>`.

4. **`<WriteupList>` row keyboard activation** — Enter or Space on a focused row navigates to `<WriteupView>` for that slug. Same as clicking. `<BracketLink href="?focus=writeups/...">` on the row provides this for free (anchor activation is keyboard-native).

6. **Focus restoration on defocus** — when the camera animates back to default orbit, focus is NOT programmatically moved. ### Mobile breakpoints (Phase 3 additions)

Phase 1 + 2's breakpoints are preserved. Phase 3 introduces NO new breakpoints. The `<MonitorOverlay>` is fixed 600×400 px regardless of viewport — drei `<Html transform>` projects this onto the 3D plane independently of CSS media queries (the projection scales via `distanceFactor` based on camera distance, not viewport width).

On a 360px-wide phone running 3D shell after `?view=3d` bypass:
- Canvas is 360px wide × ~600px tall (viewport-minus-header-minus-footer).
- The 600×400 monitor overlay still projects at the calibrated `distanceFactor` — at the default camera position the monitor renders at maybe 200×130px on the phone screen, with 14px text becoming ~5-6px text. **This is too small to read on a phone.**

- Click-to-focus on a phone DOES bring the focused monitor close enough that the 14px text becomes readable (~14px-equivalent at the focused-pose distance of ~0.65m). Recruiter who insisted on `?view=3d` on a phone CAN still read content — they just have to tap a monitor first.

  - **Auto-loading external CDN images in MDX** (`![](https://i.imgur.com/...)`) — adds runtime CDN dependency, CSP risk; all images must be in `public/assets/writeups/<slug>/`.

## Anti-Pattern Audit (Phase 3 additions)

Phase 1 + 2's anti-pattern lists apply in full. Phase 3 ADDS:

- **Type-out animation on MDX prose, About paragraph, Skills tag list, write-up index rows, code blocks, screenshot captions** → REQUIREMENTS.md OOS + Phase 1 D-07. Type-out is RESERVED for the THREE `<WhoamiGreeting>` prompt lines only. Output lines below each prompt render instantly.
- **Permanent loop on `<WhoamiGreeting>` sequence** → D-07 explicitly. Sequence runs ONCE per mount; cursor blinks indefinitely after.
- **Fake REPL inside the center monitor** (a text input below `<WhoamiGreeting>` that "accepts" commands but only handles `whoami` / `ls`) → REQUIREMENTS.md OOS (fake interactive password prompt / fake terminal that doesn't accept input). The `<WhoamiGreeting>` is a CONSTRAINED display animation, not an interactive REPL.
- **Plagiarised CTF write-ups** → Pitfall 7 + REQUIREMENTS.md OOS. Every `<ProvenanceFooter>` cites sources; the audit at content-fill checkpoint manually verifies originality.
- **Listing tools in `skills.ts` without provenance** → CNT-03 + Pitfall 7. Every skill must appear in a project description or write-up — verified manually before phase complete (D-20).
- **Self-titling as "Ethical Hacker" / "Pentester" in any write-up or content file** → Phase 1 anti-pattern, reaffirmed.
- **`mailto:` raw HTML anywhere in MDX bodies** → Phase 1 TXT-04. Email reveals only via `<EmailReveal>` (text shell hero + center-monitor `<WhoamiGreeting>`).
- **Backlit "screen glow" tinting MDX prose via partial-opacity overlays** → emissive bleed-through is the anti-use. `<MonitorOverlay>` inner `<div>` is fully opaque (`bg-bg`).
- **Custom CRT shader on monitor surfaces** → CLAUDE.md "What NOT to Use" + V2-3D-02. Phase 4 postprocessing (Bloom + Scanline + ChromaticAberration + Noise) is the v1 CRT effect; custom shaders are v2.
- **Loading spinners on `<WriteupList>` row clicks** → write-ups are bundled at build time; there is NO loading state to surface.
- **Fade animations on `<WriteupView>` ↔ `<WriteupList>` swap** → instant cut.
- **"Read more" / "Click to read" instructional copy on write-up rows** → the row IS the click target; explicit instruction is patronising.
- **Camera intro animation on `?focus=<id>` deep-link first-mount** → instant snap. Locked above.
- **HUD overlay above the Canvas with focus-state hints** ("press Esc to return", "click a monitor", "drag to look around") → the `aria-label` on the canvas describes the affordance for screen readers; sighted users discover the interaction. Locked above.
- **Lightbox / zoom-on-click on screenshots** → adds JS, animation, focus-trap complexity. Phase 3 ships static images at full inline size.
- **Auto-numbering on `fig <N>` captions** → author increments manually. Explicit > implicit.
- **Inline `<script>` or `<style>` tags inside MDX bodies** → security + maintainability anti-pattern. MDX renders content + structure; styling comes from the renderer, scripts have no place.
- **Custom Shiki theme JSON shipped in repo** → use the bundled `github-dark` theme. v2 may revisit if Phase 4 polish demands a custom palette match.
- **Per-write-up "estimated reading time" calculated at build time** → author provides `duration` in frontmatter (D-14). Calculation drift is its own bug class; explicit author-provided string is correct.
- **`<WriteupView>` opening in a new browser tab / window** → in-place mode (D-19) — the right monitor's content swaps; URL updates via `history.replaceState`. NO `target="_blank"`.
- **Modal / dialog / tooltip on top of the Canvas** → no modals in Phase 3. The `<MonitorOverlay>` is NOT a modal — it's drei `<Html transform>` projected onto the screen plane geometry; the user can still see and interact with the rest of the scene (or animate camera back via Esc).
- **`localStorage` writes for focus state** → URL is the source of truth (D-11). Refreshing the page on `?focus=projects` lands at the same focused state. NO localStorage.
- **`prefers-reduced-motion` toggle in the UI** → respect the OS setting; do not provide a UI override (browser-level setting + user-agent CSS pref is the correct surface).
