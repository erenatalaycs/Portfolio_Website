# Phase 2 — UI Design Contract

> Visual and interaction contract for Phase 2 (3D Shell + Asset Pipeline + Capability Gating). Generated by gsd-ui-researcher, verified by gsd-ui-checker. **This document EXTENDS the Phase 1 UI-SPEC; it does NOT redefine the palette, typography scale, spacing scale, or any existing component contract.** Every locked decision in `02-CONTEXT.md` (D-01..D-15) is honoured here, not re-debated.

## Inheritance Notice

The Phase 1 UI-SPEC is the canonical source of truth for:

- **Palette tokens** — eight-color `@theme` block (`--color-bg #0d1117`, `--color-fg #c9d1d9`, `--color-accent #7ee787`, `--color-warn #e3b341`, `--color-negative #ff7b72`, `--color-muted #8b949e`, `--color-surface-1 #161b22`, `--color-focus #79c0ff`).
- **Typography scale** — JetBrains Mono, 4 sizes (14/16/20/24px), 2 weights (400/600), body line-height 1.5, body max-width 72ch.
- **Spacing scale** — Tailwind v4 default 4-multiple scale; 44×44 touch-target rule on every interactive element; one documented 12px exception.
- **`[bracket]` link contract** — five states (default / hover / focus-visible / active / `aria-current`).
- **Skip-to-content link**, semantic `<header>/<main>/<footer>` shape, focus order.
- **Anti-pattern audit** — Matrix rain, padlock/shield motifs, hooded-hacker silhouettes, skill bars, type-out on body content, `#000000`+`#00ff41`, fake REPL — ALL still forbidden in Phase 2.

Phase 2 ADDS: (a) a `?view=3d|text` toggle visual + camera-mode toggle on the same line, (b) a 3D-Canvas-DOM-overlay layout, (c) a context-loss info bar, (d) a 3D-scene material/lighting/camera contract, (e) a Suspense / lazy-load behaviour spec, (f) reduced-motion handling for the 3D scene. Nothing else changes.

**Inherited verbatim from Phase 1.** Eight-token `@theme` palette is the entire Phase 2 palette — no new tokens added.

### How Phase 1 tokens map to Three.js materials (D-06)

The 3D scene reads HEX values from a single shared module `src/scene/colors.ts` that mirrors `src/styles/tokens.css`. **R3F materials cannot read CSS custom properties directly** — the values are mirrored as TS constants and a unit test asserts each constant equals the corresponding `@theme` token (verified at lint/test time so the two cannot drift):

```ts
// src/scene/colors.ts (planner authors)
export const SCENE_COLORS = {
  bg:       '#0d1117', // matches --color-bg
  surface:  '#161b22', // matches --color-surface-1
  accent:   '#7ee787', // matches --color-accent
  warn:     '#e3b341', // matches --color-warn
} as const;
```

### Accent reserved for (Phase 2 additions to Phase 1's list)

In Phase 1, accent (`#7ee787`) was reserved for 8 specific text uses. Phase 2 extends this list with **two new in-DOM uses** plus **one in-canvas use**:

9. **Active state on `<ViewToggle>` and `<CameraToggle>` `<BracketLink>` elements** — the currently-selected option (`[3d]` when in 3D shell, `[text]` when in text shell, `[orbit]` or `[free]` for camera mode) inverts: `background-color: var(--color-accent); color: var(--color-bg);`. Implemented as a new `active?: boolean` prop on `<BracketLink>` (the existing `:active` pseudo-class flash from Phase 1 is unchanged — it fires for one frame on click; the new `active` PROP holds the inverted styling persistently while the toggle option is selected). The two are visually congruent: the click-flash IS what selection looks like.
10. **`[retry 3D]` link inside the `<ContextLossBar>`** — standard `<BracketLink>`, no special variant.
11. **In-canvas:** monitor screens emissive (`#7ee787` as `emissiveIntensity={1.5}`) — D-06.

### Warn reserved for (Phase 2 additions to Phase 1's list)

In Phase 1, warn (`#e3b341`) was reserved for 1 use (cert "in progress" leading bracket). Phase 2 extends with **two new uses**:

3. **`[!]` glyph leading the `<ContextLossBar>` body text** — D-13. Single character, no underline, no link behaviour. Renders as `<span class="text-warn" aria-hidden="true">[!]</span>` followed by the actual message.
4. **In-canvas:** lamp bulb emissive (`#e3b341` as `emissiveIntensity={2.0}`) — D-06.

### NOT permitted in Phase 2 (anti-uses to police in review)

- **Accent on the camera-toggle DIVIDER element** — divider uses 20%-mixed accent (`color-mix(in srgb, var(--color-accent) 20%, transparent)`), the same opacity recipe as Phase 1's sticky-nav bottom border. Solid accent on a non-text decorative element would break the "accent reserved for voice" rule.
- **Negative (`#ff7b72`) anywhere in Phase 2.** The context-loss bar is a degraded-but-usable state, not a hard error — `[!]` warn is correct. Negative is reserved for the 404 page's `bash: cd: ... No such file or directory` line and stays Phase-1-only.
- **No new in-canvas colours.** Three.js scene uses exactly the four tokens listed above (`bg`, `surface-1`, `accent`, `warn`). No rim-light tints, no fill colours, no fog colours. Phase 4 postprocessing (Bloom + ChromaticAberration) tints the output but does not add palette tokens.
- **No accent or warn on any Suspense indicator.** If the planner adds an inline indicator next to the view-toggle while the 3D chunk loads (see "Loading state UX" below), it renders in **muted** color (`--color-muted`) — explicitly NOT accent (an indicator must not draw the eye more than the navigation).

## Copywriting Contract

### View toggle (D-09 / D-10)

The exact rendered text. Lives inside the sticky header, immediately below the existing `$ goto […]` nav line, in the SAME header `<nav>` container.

```
$ goto    [about]  [skills]  [projects]  [certs]  [writeups]  [contact]
$ view    [3d]  [text]
```

In the 3D shell, the line extends with the camera-mode toggle, separated by a vertical rule:

```
$ view    [3d]  [text]   │   $ camera  [orbit]  [free]
```

- `[3d]`, `[text]`, `[orbit]`, `[free]` are real `<button>` elements styled visually identically to `<BracketLink>`. (NOT anchors — they don't navigate to a new URL semantically; they update query-state via `history.replaceState`.) The button visual is a small additive variant of `<BracketLink>` — the planner's call whether to ship a sibling `<BracketButton>` component or accept a `as="button"` prop on `<BracketLink>`. **Recommendation:** add `as?: 'a' | 'button'` to `<BracketLink>` (default `'a'`) so the visual contract has a single source of truth. The five-state spec (default / hover / focus-visible / active-click / `active` prop) applies identically.
- The active option uses the new `active?: boolean` prop: persistent `background-color: var(--color-accent); color: var(--color-bg);` for the duration of the URL-state being that value. The bracket glyphs `[` and `]` invert with the inner label so the entire bracket-pair reads as "selected".
- The `$` prompt and the keyword (`view`, `camera`) come from `<TerminalPrompt>` — same component as the `$ goto` line. The keyword (`view` / `camera`) renders in body color (`text-fg`), matching how `goto` is rendered in Phase 1.
- **Ordering is fixed:** `[3d]` before `[text]` (3D is the differentiating feature; recruiters who want the text shell click `[text]`). `[orbit]` before `[free]` (orbit is the safer default).

### Context-loss info bar (D-13)

The exact rendered text:

```
[!] 3D scene unavailable on this device. You're on the text shell.    [retry 3D]    [×]
```

- `[!]` is a literal text character `[!]` rendered in `--color-warn` (`#e3b341`), NOT a parsed bracket glyph (it's not a link). Implementation: `<span class="text-warn font-mono" aria-hidden="true">[!]</span>`. Screen readers see the full sentence via the bar's `role="status"` live region (see Accessibility section).
- The body text `3D scene unavailable on this device. You're on the text shell.` renders in `--color-fg` (body color). Period at end, single sentence, no exclamation marks.
- `[retry 3D]` is a `<BracketLink>` (anchor) with `href` computed as `import.meta.env.BASE_URL + '?view=3d'`. Click triggers `window.location.assign(...)` — full page reload — per D-14. The HREF is visible in the status bar on hover for transparency.
- `[×]` is a button (NOT an anchor) that hides the bar. `aria-label="Dismiss notification"`. Visual: `<BracketLink>` styling with `as="button"`. The character is U+00D7 (MULTIPLICATION SIGN), which renders cleanly in JetBrains Mono.
- **Auto-dismiss:** 8 seconds after the bar mounts (D-13 / Claude's-discretion: 5-10s, lock at 8s — long enough for a recruiter to read 11 words at 200 wpm with comfortable margin, short enough that a stale bar doesn't clutter the page after the user has moved on).
- **Forbidden alternative copy** (do NOT use):
  - `Oops! Something went wrong.` — non-honest, non-terminal voice.
  - `WebGL context lost — please reload.` — too technical for recruiters; "context" is a developer term.
  - `Switching to text mode...` — implies an animation that doesn't exist (D-13 is instant cut).
  - `Click here to retry` — a `[bracket]` link is the affordance; "click here" is a 1995-web anti-pattern.

### Loading state UX (Suspense fallback while 3D chunk downloads)

Per D-01 and Claude's-discretion: **the Suspense fallback IS the actual `<TextShell />`, not a skeleton.** Recruiters who land before the 3D chunk has finished downloading get the full text shell — they may not even notice the swap if the chunk lands fast. The text shell IS the recruiter-grade product (TXT-01); a placeholder skeleton would be strictly worse.

**Inline indicator decision: NO indicator on the toggle line.** Locked here, not deferred. Rationale:

- The text shell during Suspense is fully usable; flagging it as a "loading" state would be misleading — it's THE product, not a placeholder.
- An indicator on `[3d]` would imply the toggle is currently disabled, which it isn't (clicking `[3d]` again while loading is a no-op but not an error).
- Adding muted-colour `[loading…]` text would be the first place the muted token appears in the header, and it's not load-bearing copy.

**Behaviour during the Suspense window:**
1. User clicks `[3d]` from text shell.
2. URL updates to `?view=3d` immediately via `history.replaceState`.
3. `useQueryParams` `qpchange` event fires; `<App>` decides to mount `<ThreeDShell>`; React's `<Suspense fallback={<TextShell />}>` shows the existing TextShell while `React.lazy(() => import('./shells/ThreeDShell'))` downloads the chunk.
4. The toggle's `active` prop flips to `[3d]` IMMEDIATELY (URL is the source of truth — the user's intent is now "3D"), so the inverted styling appears even though the canvas hasn't mounted yet. This is correct: the visual contract reflects intent, the rendering catches up.
5. When the chunk lands, React swaps the Suspense child in. Visual cut from TextShell to 3DShell is instant (no animation — see Animation contract).

If chunk download fails (network drop), Vite's import-on-error hook surfaces a generic error boundary — **out of scope for Phase 2's UI-SPEC**; planner picks an error message like `[!] 3D failed to load. [retry] [stay on text]` if a boundary ships, or punts to "the URL stays `?view=3d`, the user manually reloads." Treat as an edge case; do NOT design a full retry UI for it.

### First-mount camera intro animation (D — Claude's-discretion default "no")

**Locked: NO camera intro animation on first 3D mount.** The camera snaps to its default orbit position (described under "Camera contract" below) instantly when `<ThreeDShell>` mounts. Rationale:

- A dolly-in / fade-in adds 1-3 seconds of "loading feel" on top of the chunk-download already incurred.
- Phase 3's click-to-focus camera animation is the FIRST cinematic camera move (3D-06). Reserving cinematic motion for Phase 3 keeps the first-impression contract crisp: chunk-loaded → instant scene → user explores.
- Reduced-motion users get the same experience as everyone else, which means there's no separate code path to maintain.

Rejected alternatives (do NOT implement in Phase 2):
- Dolly-in from far → too theatrical, conflicts with terminal-pragmatism aesthetic.
- Fade-in opacity 0 → 1 on the Canvas → confuses the eye (text shell was just there; brief black frame reads as broken).
- Boot-up "scanline flicker" → that's the Phase 4 one-shot Glitch effect (3D-08), explicitly not Phase 2.

## Layout & Interaction Contract

### Shared `<Header>` component (Phase 2 refactor)

**Phase 1's `<StickyNav>` becomes `<Header>` and is rendered by BOTH shells.** This is the structural change the orchestrator's note flagged. The change is small:

- Rename or wrap `<StickyNav>` → `<Header>` (planner picks). The semantic element stays `<header role="banner">`.
- `<Header>` renders the existing `$ goto […]` nav line PLUS the new `$ view […] [│ $ camera […]]` toggle line.
- Both shells (`<TextShell>` and `<ThreeDShell>`) render `<Header>` as their first child. In `<TextShell>` the header is followed by `<main class="mx-auto max-w-[72ch] px-4 md:px-6 py-12">` (Phase 1 contract). In `<ThreeDShell>` the header is followed by `<main class="flex-1 min-h-0">` containing the `<Canvas>` (Phase 2 contract — see below).

### 3D shell DOM structure (3D-03)

The view-toggle DOM overlay MUST be a sibling of `<Canvas>`, not a child. Per 3D-03 and the orchestrator note. Layout:

```
<body class="bg-bg text-fg font-mono min-h-screen flex flex-col">
  <SkipToContent />                       <!-- inherited from Phase 1 -->
  <Header />                              <!-- sticky top-0 z-10, renders both prompt lines -->
  <main id="main" class="flex-1 min-h-0 relative">  <!-- 3D shell main: full remaining height -->
    <ContextLossBar />                    <!-- conditionally rendered, sits above Canvas -->
    <Canvas
      class="block w-full h-full"
      camera={{ position: [...], fov: 50 }}
      dpr={[1, 1.5]}
      shadows
      onCreated={({ gl }) => gl.canvas.addEventListener('webglcontextlost', handler)}
    >
      <Lighting />                        <!-- ambient + directional + emissives -->
      <Workstation />                     <!-- desk + monitors + lamp + bookshelf + floor -->
      <Camera />                          <!-- <PerspectiveCamera makeDefault> -->
      <Controls cameraMode={cameraMode} /> <!-- clamped or unclamped OrbitControls -->
    </Canvas>
  </main>
  <Footer />                              <!-- inherited from Phase 1 -->
</body>
```

**Key invariants:**
- `<body>` is `flex flex-col min-h-screen`. `<header>` is intrinsic-height. `<main>` is `flex-1 min-h-0` so it absorbs all remaining vertical space. `<footer>` is intrinsic-height.
- `<Canvas>` fills `<main>` (`w-full h-full block`). Canvas is the ONLY child of `<main>` in normal state — `<ContextLossBar>` is conditional and unmounts both Canvas AND itself when transitioning back to text shell.
- The header sits ABOVE the Canvas in normal DOM flow (sticky `top-0 z-10`), so it's a sibling-of-Canvas DOM overlay (3D-03). Same component renders in both shells, no fork.
- `<main>` has `position: relative` so any future absolute-positioned child (e.g. drei `<Loader>` if planner adds one) can layer over the Canvas without escaping the main bounds.
- `<Canvas>` height resolves to viewport-minus-header-minus-footer via the flexbox. On a 1024×768 viewport with a ~64px header and ~48px footer, Canvas gets ~656px — adequate for 1.2-4m camera distance to frame the whole workstation.

### Toggle component contract

#### `<ViewToggle currentView="text" | "3d">`

```
$ view  [3d]  [text]
```

- Two `<BracketLink as="button">` elements wrapped in a `<TerminalPrompt><span class="text-fg">view</span></TerminalPrompt>` prefix.
- `[3d]`: `active={currentView === '3d'}`. On click: `setQueryParams({ view: '3d' })` → triggers `qpchange` event → `<App>` re-renders with `<ThreeDShell>`.
- `[text]`: `active={currentView === 'text'}`. On click: `setQueryParams({ view: 'text' })` → triggers `qpchange` → `<App>` re-renders with `<TextShell>`.
- **No localStorage write** (D-12). URL is the single source of truth.
- Capability-check bypass: clicking `[3d]` is equivalent to typing `?view=3d` in the URL — D-02 says capability check is COMPLETELY SKIPPED on `?view=3d`. The toggle calls `setQueryParams({ view: '3d' })`, the App reads `?view=3d`, the App skips `detectCapability()`, the 3D shell mounts. No friction, no warning modal.

#### `<CameraToggle cameraMode="orbit" | "free">`

```
$ camera  [orbit]  [free]
```

- Same shape as `<ViewToggle>` (TerminalPrompt prefix + two `<BracketLink as="button">` elements).
- State is **ephemeral**, held in component-local `useState` (NOT URL, NOT localStorage — D-11). Resets to `'orbit'` on every page load.
- On change: lifts state up to `<ThreeDShell>` so `<Controls cameraMode={cameraMode}>` re-renders with the new clamping.
- `<OrbitControls>` reconfigures: `'orbit'` → mid-tier clamps (D-08); `'free'` → unclamped (full polar + azimuth + zoom).

### Camera contract (D-08)

#### Default mode: `orbit` (mid-tier clamps)

| `OrbitControls` prop | Value | Rationale |
|----------------------|-------|-----------|
| `minPolarAngle` | `Math.PI / 3` (60°) | Can look slightly down at desk surface; cannot look at the floor |
| `maxPolarAngle` | `Math.PI * (100 / 180)` (100°) | Can look slightly up at top of monitors; cannot look at the ceiling (which doesn't exist anyway, but prevents the disorienting "looking up at nothing") |
| `minAzimuthAngle` | `-Math.PI / 2` (-90°) | Can pan to the left side of the desk |
| `maxAzimuthAngle` | `Math.PI / 2` (+90°) | Can pan to the right side of the desk; cannot get behind it |
| `minDistance` | `1.2` | 1.2 metres — close enough to read details on a Phase 4 GLB but not so close the monitors clip |
| `maxDistance` | `4.0` | 4 metres — far enough to frame the whole desk + bookshelf |
| `enablePan` | `false` | No `cameraTarget.set()` style panning; orbit only |
| `enableDamping` | `true` | drei default; gives smooth feel |
| `dampingFactor` | `0.08` | Slightly more damped than drei's default (0.05) — feels like a heavy CRT, not a quadcopter |
| `target` | `[0, 0.6, 0]` | Centred on the desk surface (`y=0.6` is roughly desk-top height of 0.75m minus small offset to centre the monitors in frame) |

#### Free mode: unclamped

All `min*`/`max*` angle and distance constraints removed (drei defaults: full sphere). `enablePan` stays `false` for now (panning + free rotation lets users get utterly lost; planner may revisit if user feedback demands).

#### Default camera position

`[2.4, 1.4, 2.4]` — diagonal-front view at ~human-eye height, framing the desk + monitors with bookshelf visible in background. Looking at `[0, 0.6, 0]`. FOV `50` degrees (drei default — natural-feeling, not wide-angle).

| Prop | Value | Rationale |
|------|-------|-----------|

### `<ContextLossBar>` layout (D-13)

```html
<div role="status" aria-live="polite" class="bg-surface-1 border-b border-b-[color-mix(in_srgb,var(--color-warn)_30%,transparent)] px-4 py-2 font-mono text-fg">
  <div class="mx-auto max-w-[72ch] flex flex-wrap items-center gap-x-4 gap-y-1">
    <span aria-hidden="true" class="text-warn">[!]</span>
    <span>3D scene unavailable on this device. You're on the text shell.</span>
    <BracketLink href="?view=3d" onClick={triggerReload}>retry 3D</BracketLink>
    <BracketLink as="button" onClick={dismiss} ariaLabel="Dismiss notification">×</BracketLink>
  </div>
</div>
```

- `bg-surface-1` (`#161b22`) — the only Phase 1-permitted alt-surface, used here for the same purpose (signalling "raised content"). The bar is visually a thin slab on top of the page background.
- Bottom border: 30%-mixed warn (`color-mix(in srgb, var(--color-warn) 30%, transparent)`). Slightly heavier than the sticky-nav's 20%-mixed accent because the bar IS the focal change in the layout — needs to register as "different".
- `role="status"` `aria-live="polite"` so screen readers announce the message when it appears, without interrupting the user's current focus.
- Layout is `flex flex-wrap` — on mobile, `[retry 3D]` and `[×]` wrap to a new line if the message is too long for the viewport.
- Sits as the FIRST child of `<main>` in the text shell (after the context-loss swap). The 3D shell never shows it (when context-lost happens, we've already swapped to text shell — the bar lives in text shell only).
- **Auto-dismiss timer:** `setTimeout` with 8000ms. Cleared if the user clicks `[×]` first. Cleared when the component unmounts.
- **No fade animation.** Instant cut in (D-13). When dismissed (auto or manual), the bar unmounts instantly — no fade out either. Reduced-motion handling is automatic via this no-animation choice.

### Capability-check UI (D-01, D-02, D-03)

The capability check is silent — D-02 explicitly: no warning modal, no "are you sure" overlay. The user experience:

- Phone (mobile UA + not iPad) lands on text shell. The `[3d]` toggle is visible. They can click it to bypass.
- Desktop with WebGL2 + `deviceMemory ≥ 4` + `hardwareConcurrency ≥ 5` lands on 3D shell. The `[text]` toggle is visible. They can click it to switch.
- iPad / Galaxy Tab S (modern tablet, `hardwareConcurrency ≥ 6` + WebGL2) lands on 3D shell — D-01.
- Reduced-motion user lands on text shell (capability check defaults reduced-motion to text). They can click `[3d]` to bypass — but the 3D scene has NO non-conditional animation in Phase 2 (see Animation contract), so the bypass produces a static-feeling scene that respects their preference.

The toggle is the entire affordance for capability bypass. No "switch to advanced view" call-out, no "click for full experience" link.

### `?view=3d` URL bypass (D-02)

Per D-02: `?view=3d` SKIPS the capability check entirely. App logic:

```ts
// Pseudocode for src/app/App.tsx
const { view } = useQueryParams();

if (view === '3d') {
  // explicit 3D request — no capability check
  return <Suspense fallback={<TextShell />}><ThreeDShell /></Suspense>;
}

if (view === 'text') {
  // explicit text request
  return <TextShell />;
}

// no explicit view — capability check decides
const { pass } = detectCapability();
if (pass) {
  return <Suspense fallback={<TextShell />}><ThreeDShell /></Suspense>;
}
return <TextShell />;
```

The `?view=3d` URL parameter is the ONLY user-facing bypass. The toggle button writes to that same parameter — same code path, same bypass behaviour.

### Touch / pointer behaviour

drei `<OrbitControls>` provides touch handling out of the box. Phase 2 confirms:

- **Single-finger drag on Canvas** → orbits the camera (in clamped range when `'orbit'` mode, full range when `'free'` mode).
- **Two-finger pinch on Canvas** → zooms (within `minDistance`/`maxDistance` clamps in `'orbit'` mode, unclamped in `'free'`).
- **`touch-action: pan-y` on `<Canvas>`** — explicitly allows page-vertical-scroll to pass through the Canvas region on touch devices (so a phone user who lands in 3D can still scroll the page). drei's default `touch-action` is `none` which traps all gestures; we override.
  - Implementation: `<Canvas style={{ touchAction: 'pan-y' }}>` or via a Tailwind utility on the wrapper.
  - **Caveat:** with `touch-action: pan-y`, two-finger pinch may conflict with browser zoom. drei's gesture handling claims priority via pointer events — verify in Phase 2 testing on iOS + Android.
- **Click on Canvas** → no-op in Phase 2 (Phase 3's click-to-focus camera animation lights this up; Phase 2 ships geometry-only with no interactive monitors).
- **Hover on Canvas elements** → no-op in Phase 2 (no hover affordance until Phase 3 introduces clickable monitors).

### ARIA contract

| Element | ARIA pattern | Notes |
|---------|--------------|-------|
| `<ViewToggle>` `[3d]` button | `aria-pressed={currentView === '3d'}`; `aria-label="Switch to 3D view"` (or just text content "3d" — recruiter-friendly, no abbreviation expansion needed) | Toggle button pattern. `aria-pressed` is the canonical attr for two-state buttons; supersedes `aria-current`. The active styling (inverted bg) provides a redundant visual cue for users who don't see the announcement. |
| `<ViewToggle>` `[text]` button | `aria-pressed={currentView === 'text'}`; `aria-label="Switch to text view"` | Same pattern. |
| `<CameraToggle>` `[orbit]` button | `aria-pressed={cameraMode === 'orbit'}`; `aria-label="Camera mode: orbit (clamped)"` | Same. The aria-label adds the parenthetical that the visible label doesn't fit, helping screen-reader users understand the difference. |
| `<CameraToggle>` `[free]` button | `aria-pressed={cameraMode === 'free'}`; `aria-label="Camera mode: free (unclamped)"` | Same. |
| `<ContextLossBar>` wrapper | `role="status" aria-live="polite"` | Screen readers announce when the bar mounts. `polite` does not interrupt; `assertive` would steal focus from active typing — wrong choice for this severity level. |
| `<ContextLossBar>` `[!]` glyph | `aria-hidden="true"` | The `[!]` is decorative; the message itself carries the warning intent. |
| `<ContextLossBar>` `[×]` button | `aria-label="Dismiss notification"` | Visible label is `×`, which is meaningless to screen readers. |
| `<Canvas>` element | `aria-label="Interactive 3D workstation scene. Drag to look around."` | drei doesn't add this; planner adds it on the `<canvas>` element via `<Canvas onCreated={({ gl }) => { gl.domElement.setAttribute('aria-label', '...'); }}>` or via styled-prop. **Alternative:** add `<noscript>` text inside the Canvas container with section-by-section parity to the text shell — defers to Phase 3 where actual content lands; Phase 2 ships geometry only, so the aria-label is sufficient. |

**Reasoning for `aria-pressed` over `aria-current`:** `aria-current` implies "this thing represents the current page/section/step in a navigation flow." The view toggle is a binary state switch, not a step-marker — `aria-pressed` is canonically correct. (Phase 1's `[bracket]` nav links use `aria-current="page"` for the section currently in view — that's a different pattern and is preserved.)

## Anti-Pattern Audit (Phase 2 additions)

Phase 1's anti-pattern list applies in full. Phase 2 ADDS:

- **Permanent lamp blink** → D-decisions (subtle blink only if reduced-motion is OFF, and even then planner-discretion / default-no). A blinking light on every page load is a junior-portfolio cliché.
- **Camera intro dolly / fade-in / "boot up" sequence on first 3D mount** → locked above ("Camera intro animation"). Phase 4 has a one-shot Glitch effect for the boot-up moment; Phase 2 does NOT.
- **"Switch to 3D for the full experience" call-out copy near the toggle** → the toggle IS the call-out; adding promotional copy reads as "begging the user to engage" (REQUIREMENTS.md "Out of Scope" excludes hire-me modals; this is the same anti-pattern in another shape).
- **A warning modal on `?view=3d` capability bypass** → D-02 explicitly forbids. "Are you sure?" friction breaks the trust contract with power users.
- **Walls / ceiling / sky in the workstation scene** → D-05 explicitly. "Hacker workstation" reads correctly as a desk floating in dark space; a full room costs polygon budget for no narrative gain.
- **Textures on procedural primitives** → D-06 explicitly flat materials. Textures land in Phase 4 with the GLB.
- **In-canvas text via drei `<Text>`** → deferred to Phase 3 with drei `<Html>` integration. Mixing `<Text>` (texture-baked SDF) with `<Html>` (DOM overlay) creates two type pipelines for one design — pick `<Html>` in Phase 3 and stick with it.

- **Permanent Glitch / DOF / SSAO postprocessing effect** → CLAUDE.md "What NOT to Use" + REQUIREMENTS.md (3D-08). Even though postprocessing is Phase 4, document the prohibition here so it doesn't accidentally land in a Phase 2 plan.
