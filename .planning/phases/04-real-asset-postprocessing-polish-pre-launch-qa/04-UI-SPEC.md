# Phase 4 — UI Design Contract

> Visual and interaction contract for Phase 4 (Real Asset, Postprocessing, Polish, Pre-Launch QA — v1.0 ship). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Phase 4 RE-USES Phase 1's design system end-to-end** (Tailwind v4 `@theme` tokens, off-black + softened-green palette, JetBrains Mono, BracketLink/TerminalPrompt primitives). Phase 4 introduces ZERO new design tokens, ZERO new typography steps, ZERO new accent colors. The four locked surfaces this contract pins are (1) the Web3Forms `<ContactForm />`, (2) the Live profiles sub-list (TryHackMe + HackTheBox), (3) the postprocessing pipeline visual contract, and (4) the OG image + JSON-LD + sitemap markup surfaces. Plus: two pre-launch checklist files (CHECKLIST-OPSEC.md extension + CHECKLIST-LAUNCH.md new).

## Spacing Scale

Phase 4 reuses Phase 1's Tailwind v4 default spacing scale verbatim. No new tokens.

| Token | Value | Phase 4 Usage (NEW or extending Phase 1) |
|-------|-------|------------------------------------------|
| xs (`p-1`) | 4px | Inline padding around `[bracket]` glyphs (existing) |
| sm (`p-2`) | 8px | Vertical gap between adjacent ContactForm fields' label-and-input pairs; gap between Live-profiles `[TryHackMe profile]` `[HackTheBox profile]` brackets |
| md (`p-4`) | 16px | Default ContactForm internal padding; gap between the `<label>` and its `<input>`; vertical rhythm between form rows; padding inside the success / failure terminal-block replacement region |
| lg (`p-6`) | 24px | Padding inside the form's outer container; vertical gap between the form and the surrounding section copy |
| xl (`p-8`) | 32px | (existing — inter-section gap; Phase 4 does not introduce new section breaks) |
| 2xl (`p-12`) | 48px | (existing) |
| 3xl (`p-16`) | 64px | (existing) |

**Exceptions:**

- **Touch target minimum 44×44px** on every ContactForm submit button, every BracketLink (`[send another]`, `[retry]`, `[TryHackMe profile]`, `[HackTheBox profile]`), and the form's `<input>` / `<textarea>` controls. Inputs achieve this via `py-3 px-3` (12px vertical × 12px horizontal padding); the BracketLink's existing `py-2 px-3 -mx-1` already satisfies the rule (Phase 1 § Spacing). 12px vertical padding on form inputs is the documented exception to the 4-multiple rule (single tightening to keep the form compact at the bottom of a scrollable monitor in 3D shell — same justification as Phase 1's 12px `mt-3` heading-to-body exception).
- **Honeypot field (`botcheck`):** uses `display: none` (NOT `visibility: hidden`, NOT `opacity: 0`, NOT `position: absolute; left: -9999px`). Per the objective brief and accessibility-tools-treat-as-absent rule. `display: none` removes the input from the accessibility tree entirely — desired behaviour because legitimate humans must NEVER see or tab to it.

### NOT permitted for accent (anti-uses to police in review):

- Bloom postprocessing in 3D scene tints emissive geometry but does NOT add a new accent palette token — Bloom reads colors from the existing `--color-accent` (monitor screens) and `--color-warn` (lamp bulb) tokens already in `src/scene/colors.ts`. **No new emissive colors introduced by Phase 4.**

### Postprocessing Pipeline Color Contract (3D-08 visual contract)

The four-effect pipeline (Bloom + Scanline + ChromaticAberration + Noise) is gated by `<PerformanceMonitor>` per CONTEXT D-05 / D-06. **Effects do not introduce new palette tokens** — they tint the scene the GPU already renders.

| Effect | Tuning | Visual Contract | Color Contract |
|--------|--------|-----------------|----------------|
| Bloom | `threshold={0.6} intensity={0.6} mipmapBlur luminanceSmoothing={0.025}` (CONTEXT D-08) | Pulls just the emissive monitor green and lamp amber — selective, not a global glow. Gives the "screen-on" CRT feel without washing out body text on the `<Html>` overlays. | Reads from existing `--color-accent` (monitor emissive) and `--color-warn` (lamp emissive). No new palette tokens. |
| Scanline | `density={1.25} opacity={0.15}` (verbatim from REQUIREMENTS 3D-08) | Static fixed-density CRT-style horizontal lines across the entire viewport. **Static — does not move, does not animate.** Reads as a CRT-overlay, not as motion. | Black scanlines at 15% opacity over the rendered scene. No palette token introduced. |
| ChromaticAberration | `offset={[0.0008, 0.0008]}` (REQUIREMENTS 3D-08) | Subtle red/blue channel separation at high-contrast edges — gives the "old monitor" optical-imperfection feel. **Static — does not animate.** | Pure GPU effect; no palette involvement. |
| Noise | `opacity={0.04}` (REQUIREMENTS 3D-08) | Low-density grain across the rendered output. At 0.04 opacity it reads as **static grain texture, not as motion**. | Pure GPU effect; no palette involvement. |

**Tier-flip visual contract (CONTEXT D-07):** When `<PerformanceMonitor>` flips `'high' → 'low'`, the four effects unmount IMMEDIATELY — no fade-out, no transition, no opacity ramp. When `'low' → 'high'` (rare; usually one-way `high → low`), effects mount immediately. Aligns with Phase 2 D-13 instant-cut precedent and Phase 1 § Animation contract (no fade animations gated by reduced-motion).

**Reduced-motion compatibility:** All four effects are STATIC (no movement, no temporal animation). Therefore `prefers-reduced-motion: reduce` does NOT need to disable them. Verified per CONTEXT § Specifics: "Bloom is static (no movement)... Scanline is also static (it's a fixed CRT-style overlay, not a moving line)... Noise opacity 0.04 is so low it reads as static grain, not movement."

**Forbidden postprocessing effects (carrying CLAUDE.md "What NOT to Use" + REQUIREMENTS 3D-08 + Phase 2 D anti-features):**
- **DOF (Depth of Field)** — blurs the very content (text on monitors) recruiters are trying to read.
- **SSAO** — expensive, marginal visual gain in a dark scene.
- **Pixelation** — destroys text readability.
- **DotScreen** — too aggressive; reads as halftone print, not CRT.
- **Permanent Glitch** — looks cool for 2 seconds, then nauseates the visitor (CLAUDE.md). Phase 4 does NOT use the boot-up `Glitch` effect either; the v1 ship is the four locked effects only.

### Honeypot — `botcheck` (CONTEXT D-11)

```html
<div style="display: none" aria-hidden="true">
  <label>
    <!-- honeypot: legitimate humans never see this; bots fill all fields -->
    Don't fill this out if you're human:
    <input type="text" name="botcheck" tabindex="-1" autocomplete="off" />
  </label>
</div>
```

- Field NAME: `botcheck` (CONTEXT D-11; `phone_hidden` and `_gotcha` were both listed as Claude's-discretion alternatives — Phase 4 picks `botcheck` for clarity in code review).
- Hide method: `display: none` on the wrapping `<div>`. Removes the field from the accessibility tree entirely — accessibility tools treat as absent. (Per the objective brief — `display: none`, NOT `visibility: hidden`.)
- `tabindex="-1"` keeps the field unreachable by keyboard even if a stylesheet override re-displays it.
- `autocomplete="off"` prevents browser fill from accidentally populating it.
- **On submit, if `botcheck` is non-empty:** the form abort silently — re-render the success state UI (do NOT reveal that the form caught a bot, per CONTEXT § Specifics "silent success on bot detection — defeats the bot's retry logic"). No network call is made.

### Success state (CONTEXT D-12) — verbatim copy

On 200-OK Web3Forms response, the form `<form>` element unmounts and a terminal block replaces it in-place (same DOM region; no layout shift):

```
$ message_sent
Delivered to eren.atalay@…  — I'll reply within 48h.
[send another]
```

- `$ message_sent` is a `<TerminalPrompt>` line (accent `$` glyph + body-color command text — except the command text here uses the heading 20px size to give the success state weight).
- `Delivered to eren.atalay@…` — the `…` is a literal U+2026 horizontal ellipsis character (NOT three periods). The address is partially redacted by design — it's a confirmation of WHERE the message went, but it does NOT re-leak the full obfuscated address that Phase 1's `<EmailReveal>` defends. Show the local-part-prefix only: `eren.atalay@…`.
- `— I'll reply within 48h.` — the em-dash is a literal U+2014. 48h is the contract Eren sets at v1.0 launch; if Eren wants to change it, edit `src/ui/ContactForm.tsx` (one source of truth, not duplicated).
- `[send another]` is `<BracketLink as="button">` — `onClick` resets the form state (clears `name`, `email`, `message` to empty, re-mounts the `<form>` element).

### Failure state (CONTEXT D-12) — verbatim copy

On non-2xx Web3Forms response (network error, 4xx, 5xx — anything that isn't a clean 200):

```
[!] $ message_failed
Network error. Try email instead: [eren.atalay@… (click to reveal)]
[retry]
```

- The leading `[!]` glyph is warn amber — same semantic as Phase 2's context-loss bar `[!]` (something went wrong, here's a path forward, this is not destructive).
- `$ message_failed` uses the same heading-weight terminal-prompt treatment as `$ message_sent`.
- `Network error.` — body prose, body color. **Generic copy on purpose** — DO NOT leak the actual HTTP status code or the Web3Forms server error string (information-disclosure surface; recruiters don't need debugging info).
- `[eren.atalay@… (click to reveal)]` — reuses the existing `<EmailReveal>` component (Phase 1 TXT-04), passing `identity.emailEncoded` from `src/content/identity.ts`. Same visual contract: pre-click button label, on-click decode + clipboard write + `(copied)` toast.
- `[retry]` is `<BracketLink as="button">` — `onClick` re-submits the form with the existing field values (does NOT clear the form; the user's typed message is preserved across retries).

## Layout & Interaction Contract

### `<ContactForm />` layout (the new component this phase ships)

**Outer container:**

```jsx
<div className="mt-6 p-6 border border-surface-1 rounded-none font-mono">
  {/* form contents */}
</div>
```

- `border border-surface-1` — 1px solid surface-1 frame around the form region. Reads as "a panel inside the page" without changing the bg fill.
- `rounded-none` — sharp corners. Terminal aesthetic does not use rounded corners (carrying Phase 1's design intent — no rounded corners anywhere in v1).
- `mt-6` (24px) above — gap between the existing email/GitHub/LinkedIn strip in `<Contact />` and the new form.
- `p-6` (24px) interior padding — comfortable but compact, fits inside the 600px-wide center monitor `<Html>` overlay.

**Form structure (semantic HTML):**

```jsx
<form
  onSubmit={handleSubmit}
  noValidate={false}
  className="flex flex-col gap-4"
>
  <h2 className="text-xl font-semibold">
    <TerminalPrompt><span className="text-fg">cat send_message.md</span></TerminalPrompt>
  </h2>
  <p className="text-fg">Send a message:</p>

  <label className="flex flex-col gap-1">
    <span className="text-fg">name <span className="text-accent" aria-hidden="true">*</span></span>
    <input
      name="name"
      type="text"
      required
      maxLength={80}
      className="bg-transparent text-fg border-0 border-b border-b-surface-1 focus-visible:border-b-2 focus-visible:border-b-accent py-3 px-3 outline-none"
    />
    <span className="text-muted text-sm"># max 80 chars</span>
  </label>

  <label className="flex flex-col gap-1">
    <span className="text-fg">email <span className="text-accent" aria-hidden="true">*</span></span>
    <input name="email" type="email" required pattern=".+@.+\..+" className="..." />
    {emailError && <span className="text-warn text-sm"># invalid email</span>}
  </label>

  <label className="flex flex-col gap-1">
    <span className="text-fg">message <span className="text-accent" aria-hidden="true">*</span></span>
    <textarea name="message" required maxLength={2000} rows={6} className="..." />
    <span className="text-muted text-sm"># max 2000 chars · {used} used</span>
  </label>

  {/* Honeypot — display: none, never tab-reachable */}
  <div style={{ display: 'none' }} aria-hidden="true">
    <label>
      Don't fill this out if you're human:
      <input name="botcheck" type="text" tabIndex={-1} autoComplete="off" />
    </label>
  </div>

  {/* Hidden access key (Web3Forms requirement) */}
  <input type="hidden" name="access_key" value={import.meta.env.VITE_WEB3FORMS_KEY} />

  {/* Hidden subject — interpolated client-side per CONTEXT D-11 */}
  <input type="hidden" name="subject" value={`[Portfolio enquiry] from ${name || '(name empty)'}`} />

  <BracketLink as="button" onClick={handleSubmit}>
    {submitting ? 'Sending…' : 'Send message'}
  </BracketLink>
</form>
```

- **Field validation:** HTML5 `required` + `pattern=".+@.+\..+"` on the email input + `maxLength` on name/message. NO zod, NO react-hook-form, NO vee-validate. (Claude's-discretion in CONTEXT — locked.)
- **Email validation timing:** `onBlur` (not on every keystroke) — typing-while-validating reads as nagging. Show `# invalid email` only AFTER the user blurs the email field with an invalid value.
- **Form submit:** `e.preventDefault()` → check honeypot → if non-empty, render success state silently → otherwise `fetch('https://api.web3forms.com/submit', { method: 'POST', body: new FormData(form) })` → on 200 render success state; on non-2xx render failure state.
- **Disabled state during submit:** all `<input>` / `<textarea>` get `disabled` attribute; submit button label changes to `[Sending…]`; user cannot click submit twice. Re-enabled on success/failure render.

### Postprocessing pipeline integration (3D-08)

Per CONTEXT D-09: `<EffectComposer>` mounted as a Suspense-deferred sibling of the workstation under `perfTier='high'`. Lazy-loaded `@react-three/postprocessing` chunk — low-tier devices NEVER download postprocessing JS.

```jsx
// src/3d/EffectComposer.tsx (NEW)
import { Suspense, lazy } from 'react';
import { PerformanceMonitor } from '@react-three/drei';

const PostFX = lazy(() => import('./PostFX')); // EffectComposer + Bloom + Scanline + CA + Noise

export function ScenePostprocessing() {
  const [tier, setTier] = useState<'high' | 'low'>('high');
  return (
    <>
      <PerformanceMonitor
        bounds={() => [30, 50]}
        onIncline={() => setTier('high')}
        onDecline={() => setTier('low')}
      />
      {tier === 'high' && (
        <Suspense fallback={null}>
          <PostFX />
        </Suspense>
      )}
    </>
  );
}
```

- Tier-flip is INSTANT — no animation, no opacity ramp (CONTEXT D-07).
- Suspense fallback = `null` (no visual indicator while postprocessing chunk loads — the scene is already perfectly readable without effects; the "no effects" intermediate state IS the low-tier fallback rendered intentionally).
- The `<PerformanceMonitor>` `flipflops`, `iflag`, and exact bounds tuning is Claude's discretion (CONTEXT § Claude's Discretion); planner picks per drei recommended pattern.
- **EffectComposer `multisampling={0}`** (postprocessing default) per CONTEXT § Claude's Discretion. If Bloom quality reads degraded on real GLB, planner escalates as a deviation.

## Anti-Pattern Audit (must-not-build)

This list extends Phase 1's anti-pattern audit. Each item references its source so it cannot be re-added on a whim.

**Carried from Phase 1 (still applies in Phase 4):**
- Matrix code rain, padlock motifs, hooded-hacker silhouettes, skill bars, type-out animation on body content, pure `#000000` + `#00ff41`, fake REPL, raw `mailto:` HTML, auto-playing audio, hire-me modals, custom cursor.

**Phase 4-specific additions:**
- **Permanent `<Glitch />` postprocessing effect** → CLAUDE.md "What NOT to Use" (`Glitch` looks cool for 2 seconds, then nauseates). Phase 4 does NOT use Glitch even for the hero boot-up — the four locked effects (Bloom + Scanline + CA + Noise) are the entire postprocessing budget.
- **`<DepthOfField>` postprocessing** → CLAUDE.md (blurs the very content recruiters are reading). REQUIREMENTS 3D-08 explicitly forbids DOF.
- **`<SSAO>` postprocessing** → CLAUDE.md (expensive, marginal in dark scene). REQUIREMENTS 3D-08 explicitly forbids SSAO.
- **`<Pixelation>` postprocessing** → CLAUDE.md (destroys text readability).
- **`<DotScreen>` postprocessing** → CLAUDE.md (halftone print, not CRT).
- **A spinner glyph in `[Sending…]`** → typography contract (only one font weight + size set; no glyph not in JetBrains Mono Latin subset). The literal `…` U+2026 is the only allowed loading-state visual.
- **Gradient backgrounds anywhere** → Phase 1 § Color: "the entire site has exactly one background color (`#0d1117`)". Phase 4 introduces zero new fills.
- **A `<dialog>` confirmation modal for form submit / send-another** → Phase 1 § Destructive confirmations (no destructive actions in v1.0; submit is creative; reset is layout-only).
- **A "Coming soon" stub for missing TryHackMe or HackTheBox profile** → CONTEXT D-13 explicit ("missing one is omitted; no empty placeholder").
- **A toast / banner / popup announcing "v1.0 just shipped!"** → Phase 1 § OOS (no hire-me / newsletter modals; same anti-pattern energy).
- **Auto-playing the GLB's idle camera tour** → Phase 4 does not introduce new camera motion beyond Phase 3's click-to-focus. The default camera pose on 3D-shell mount is the same orbit pose as Phase 2 D-04.
- **A "rate this portfolio" button / link** → CONTEXT D-15 ("Anonymous public-sub feedback is rejected... low signal-to-noise; Pitfall 7 junior authenticity trap"). Peer review happens off-site, not via a UI surface.
- **Plaintext email in JSON-LD `<script>`** → STATE.md `[Phase ?] Plan 01-06: JSON-LD Person omits email field per Pitfall 4`. Phase 4 verifies the omission survives.
