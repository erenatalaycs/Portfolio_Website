# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1 (Foundation + 2D Recruiter-Grade Shell). Generated by gsd-ui-researcher, verified by gsd-ui-checker. Pins every open visual + interaction decision the planner and executor need; every locked decision in `01-CONTEXT.md` (D-01..D-18) is honoured here, not re-debated.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (raw Tailwind v4 with `@theme` tokens — single-page custom terminal aesthetic; shadcn not applicable in Phase 1: no dialogs, dropdowns, or complex form controls; first shadcn-eligible widget is the Phase 4 Web3Forms contact form, decision deferred to that phase) |
| Preset | not applicable |
| Component library | none (semantic HTML + Tailwind utility classes; shared `src/ui/*` primitives authored in-repo) |
| Icon library | none in Phase 1 — terminal aesthetic uses bracket-glyph affordances (`[CV]`, `[ ]`, `[✓]`, `$`) instead of icons; if any icon is unavoidable later, use [`lucide-react`](https://lucide.dev/) (tree-shakeable, neutral line style — defer the decision until Phase 2 where the view-toggle DOM overlay may need it) |
| Font | **JetBrains Mono** via `@fontsource/jetbrains-mono` (self-hosted, woff2-only, Latin subset, weights 400 + 600). Per D-08. **No Google Fonts call.** Single font payload target ≤ 60 KB. System monospace fallback chain: `'JetBrains Mono', ui-monospace, 'SF Mono', 'Menlo', 'Consolas', monospace`. |

## Spacing Scale

Declared values (multiples of 4, mapped to Tailwind v4 default spacing tokens — no custom tokens needed for Phase 1):

| Token | Value | Usage |
|-------|-------|-------|
| xs (`p-1`) | 4px | Inline padding around bracket tags (`[python]`), bracket-link hit-area expansion |
| sm (`p-2`) | 8px | Gap between adjacent skill tags / nav items; vertical rhythm inside dense terminal blocks |
| md (`p-4`) | 16px | Default block-level padding inside section bodies; line-height-ish gap between `whoami` lines |
| lg (`p-6`) | 24px | Padding inside the sticky-nav bar; spacing between hero block paragraphs |
| xl (`p-8`) | 32px | Inter-section gap between adjacent `<section>` blocks |
| 2xl (`p-12`) | 48px | Top padding above section headings; bottom padding above the footer |
| 3xl (`p-16`) | 64px | Hero block top padding (above-the-fold breathing room) on `md+` viewports |

**Exceptions:**
- **Touch target minimum 44×44px** — every `[bracket]` link in the sticky nav and hero must have an effective click/tap area ≥ 44×44 (achieved via `py-2 px-3` on the wrapping anchor, NOT by enlarging the visible bracket glyph). Required by WCAG 2.5.5 (Target Size) and by the mobile-wraps-onto-multiple-lines rule from D-05.
- **Section heading (`$ cat about.md`) → first body paragraph** uses tighter spacing: `mt-3` (12px), not the default `mt-4` — the prompt and its output should read as a single terminal stanza, not two disconnected blocks. 12px is a deliberate exception to the 4-multiple rule applied as a single tightening; encoded as a Tailwind utility, no custom token.

## Typography

JetBrains Mono throughout. Two weights only (regular 400, semibold 600). Three body/label sizes + one heading size = 4 total scale steps. Body line-height ≥ 1.5 (D-decisions / Claude's-discretion section).

| Role | Size | Weight | Line Height | Notes |
|------|------|--------|-------------|-------|
| Body | 16px (`text-base`) | 400 | 1.5 (`leading-relaxed`) | Default for `<p>`, project descriptions, About paragraph, certification entries, MDX prose. |
| Label / inline tags | 14px (`text-sm`) | 400 | 1.4 (`leading-snug`) | Skill tags `[python]`, project metadata (date / repo URL beside title), footer copyright. |
| Heading (section + hero) | 20px (`text-xl`) | 600 | 1.3 (`leading-snug`) | Both the hero `$ whoami` line and section headings (`$ cat about.md`, `$ ls projects/`) — visually equal-weight; section hierarchy comes from spacing + the prompt prefix, not from a font-size step. |
| Display (404 page only) | 24px (`text-2xl`) | 600 | 1.3 (`leading-snug`) | Used solely for the 404 `bash: cd: …` line so the not-found state reads as the loudest text on the page. Nowhere else in Phase 1. |

**Body max-width / line length:** **72ch** (`max-w-[72ch]` on the article container). 70-80ch is the standard monospace reading window per the Claude-discretion notes; 72ch matches GitHub's prose width and gives ~80 chars of content with comfortable margin. Apply at the `<main>` / article-wrap level, not per-paragraph.

**No type-out animation on body content** (D-07). The single permitted animated glyph in Phase 1 is the static-or-blinking cursor `█` at the END of the hero `$ whoami` line — see Animation contract below.

## Color

**Background-first 60/30/10 doesn't map cleanly to a terminal aesthetic** — almost the entire viewport is the same dark surface, with content rendered as text on it. The split below is therefore **dominant surface / muted prose / accent voice / state colors**, all verified WCAG-AA against the dominant surface.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (surface) | `#0d1117` | Page background — the only background color. Honest off-black, GitHub-canvas-equivalent. **Not** pure `#000000` (REQUIREMENTS.md "Out of Scope"). |
| Body prose (60% of text) | `#c9d1d9` | Default body copy, About paragraph, project descriptions, education entries, footer text. Contrast vs bg: **12.26:1** (AAA). |
| Accent — terminal voice (10% of text) | `#7ee787` | The shell prompt `$`, every `[bracket]` token (sticky-nav links, hero contact links, skill tags, project list bullets), section headings, the `$ whoami` line, the cursor block `█`. Contrast vs bg: **12.32:1** (AAA). |
| Warn — in-progress | `#e3b341` | Reserved for the cert "in progress" leading bracket and label (`[ ] OSCP — in progress, target Jul 2026`). NOT used for hover, NOT used for emphasis, NOT used decoratively. Contrast vs bg: **9.72:1** (AAA). |
| Negative | `#ff7b72` | Reserved for honest negatives only — currently used by ONE element in Phase 1: the 404 page's `bash: cd: <path>: No such file or directory` line. NOT used for hover, errors that aren't Phase 1, or destructive confirmations (no destructive actions in Phase 1). Contrast vs bg: **7.51:1** (AAA). |
| Muted | `#8b949e` | Secondary metadata: cert dates (`earned May 2025`), project repo URL hint, footer "last updated" string, the `<dim>` text after the section heading. Contrast vs bg: **6.15:1** (AA body, fails AAA — acceptable for non-prose metadata at 14px+). |
| Surface raised (hover only) | `#161b22` | Background applied ONLY on hover/focus of project rows and cert rows to communicate interactivity. 1.09:1 against bg — deliberately subtle; it pairs with a left-border accent (see below) to provide the actual affordance. |
| Focus ring | `#79c0ff` | A distinct **blue** so keyboard focus is unmistakably different from the green-accent palette. 9.73:1 vs bg. Implemented as `outline: 2px solid #79c0ff; outline-offset: 2px;` (or `focus-visible:ring-2 focus-visible:ring-[#79c0ff] focus-visible:ring-offset-2 focus-visible:ring-offset-[#0d1117]`). NEVER suppressed. |

### Accent reserved for (explicit list — every single use of `--color-accent` is on this list):

1. The `$` prompt glyph at the start of every prompt line (hero, sticky-nav, section heading, 404).
2. Each `[bracket]` token in the sticky-nav (`[about] [skills] [projects] [certs] [writeups] [contact]`).
3. The four hero contact links inside `[CV] [GitHub] [LinkedIn] [email]` — bracket glyph and the inner label text both render in accent green.
4. Section headings (the entire prompt-styled line, e.g. `$ cat about.md`).
5. Skill tags `[python]`, `[bash]`, `[wireshark]`, etc.
6. The "earned" leading bracket on certifications: `[✓]`.
7. The hero cursor block `█` immediately after the `$ whoami` line.
8. Anchor underline color on body links inside MDX prose (Phase 1 only links: GitHub repo URLs in the `Project` rows, if any) — text remains body color, the underline draws in accent.

### NOT permitted for accent (anti-uses to police in review):
- Body paragraph text — body uses `#c9d1d9`, never accent green.
- Hover state on links — hover only changes underline thickness or adds a leading `>` glyph, never recolors to accent.
- Background fills — the entire site has exactly one background color (`#0d1117`); the only exception is the `#161b22` raised-surface hover state.
- The `[ ]` empty bracket on in-progress certs (those use the **warn** color, not accent).

### Tailwind v4 `@theme` token encoding

```css
/* src/styles/tokens.css */
@import "tailwindcss";

@theme {
  --color-bg:        #0d1117;
  --color-fg:        #c9d1d9;
  --color-accent:    #7ee787;
  --color-warn:      #e3b341;
  --color-negative:  #ff7b72;
  --color-muted:     #8b949e;
  --color-surface-1: #161b22;
  --color-focus:     #79c0ff;

  --font-mono:       'JetBrains Mono', ui-monospace, 'SF Mono', 'Menlo', 'Consolas', monospace;
}
```

These exact eight color tokens are the entire Phase 1 palette. No additional shades. No `bg-accent/20` opacity-derived variants in Phase 1 (Phase 2's view-toggle overlay may add one).

## Copywriting Contract

### Hero block (D-02 / D-03)

The exact rendered text. `[bracket]` tokens are real `<a>` tags, not styled spans. Newlines preserved as-is via `<pre>` or block-level rendering. No type-out animation in Phase 1 (animated `whoami` is Phase 3 / 3D-07 — keep this markup structure compatible so Phase 3 can wrap it in a typing animation without rewriting).

```
$ whoami
Eren Atalay — Junior Cybersecurity Analyst (UK)

$ links --all
[CV]  [GitHub]  [LinkedIn]  [email (click to reveal)]
```

- `[CV]` → `<a href="/Portfolio_Website/assets/Eren-Atalay-CV.pdf" download>` (href uses `import.meta.env.BASE_URL` so dev + prod both work).
- `[GitHub]` → `<a href="https://github.com/eren-atalay" target="_blank" rel="noopener noreferrer">`.
- `[LinkedIn]` → `<a href="https://www.linkedin.com/in/<handle>" target="_blank" rel="noopener noreferrer">`. Real handle filled from `src/content/identity.ts`.
- `[email (click to reveal)]` → `<button>` (NOT an anchor) that, on click, calls `revealEmail()` from `src/lib/obfuscate.ts`, replaces the button with a real `<a href="mailto:...">` populated at runtime, and copies to clipboard. Pre-click button label is literally `[email (click to reveal)]`. Post-click anchor label is the decoded address inside brackets, e.g. `[eren@example.com]`.

### Sticky nav (D-04)

```
$ goto  [about]  [skills]  [projects]  [certs]  [writeups]  [contact]
```

- Each `[bracket]` is an in-page anchor: `[about]` → `<a href="#about">`. Real `<a>` tags with `href`, no JS onClick handlers required for navigation.
- The string and the section ordering are imported from a single const `SECTIONS` in `src/content/sections.ts` — same const consumed by 404 page sitemap (D-14) and Phase 2's 3D camera-focus presets.

### Section headings

Prompt format: `$ cat <slug>.md` for prose-style sections, `$ ls <slug>/` for list-style sections. Exact strings:

| Section | Heading copy | Anchor id |
|---------|--------------|-----------|
| About | `$ cat about.md` | `#about` |
| Education | `$ cat education.md` | `#education` |
| Skills | `$ ls skills/` | `#skills` |
| Projects | `$ ls projects/` | `#projects` |
| Certifications | `$ ls certs/` | `#certs` |
| Write-ups | `$ ls writeups/` | `#writeups` |
| Contact | `$ cat contact.md` | `#contact` |

### Empty / placeholder states

| State | Copy |
|-------|------|
| Empty state (Write-ups, D-11 stub) | `# No write-ups published yet — first lands during Phase 3.` |
| Empty state (Projects, only if zero ready) | `# No projects published yet — see [GitHub] for ongoing work.` |
| In-progress cert | `[ ] <cert name> — in progress, target <Mon YYYY>` (warn color leading bracket; label in body color; target date is mandatory per D-10/D-12 — "studying since 2022 without a target" reads as drift) |
| Planned cert | `[ ] <cert name> — planned` (warn color leading bracket, body color label) |
| Earned cert | `[✓] <cert name> — earned <Mon YYYY>` (accent color leading bracket and `earned`; body color label; date in muted color) |
| Honest in-progress project (D-13) | `[ ] home-lab — Splunk + pfSense + Wireshark, write-up planned` (warn color leading bracket, body color description) |

### Error / 404 state (D-14)

```
$ cd /Portfolio_Website/<requested-path>
bash: cd: <requested-path>: No such file or directory

Try one of:
  $ goto  [about]  [skills]  [projects]  [certs]  [writeups]  [contact]

or [home]
```

- The `bash: cd: <path>: No such file or directory` line uses the **negative** color (`#ff7b72`) — the only Phase 1 use of negative.
- The `$ goto …` line is rendered by the SAME component as the sticky-nav, importing from the same `SECTIONS` const, so the 404 sitemap can never drift from the live site (D-14).
- `[home]` link goes to `/Portfolio_Website/` (use `import.meta.env.BASE_URL`).

### Footer

```
© 2026 Eren Atalay — built solo · source on [GitHub]
```

- Year auto-derived (`new Date().getFullYear()`) so the footer can't go stale.
- `[GitHub]` is the link to the portfolio's own repo (NOT Eren's profile — that's already in the hero).
- Rendered in muted color (`#8b949e`); accent green only on the `[GitHub]` token.

### Destructive confirmations / forms

**No destructive actions and no forms in Phase 1.** Email reveal is a single-step disclosure with no confirmation. Web3Forms contact form lands in Phase 4 (CTC-01) with its own UI-SPEC. Empty row in the table below to make this explicit.

| Element | Copy |
|---------|------|
| Primary CTA | (no CTA in Phase 1 — content is the CTA; the four hero links ARE the conversion surface) |
| Empty state heading | `# No write-ups published yet` |
| Empty state body | `# No write-ups published yet — first lands during Phase 3.` |
| Error state | (404 only, copy above) |
| Destructive confirmation | n/a — no destructive actions in Phase 1 |

## Layout & Interaction Contract

This section pins the open layout and interaction details that the planner needs but the spacing/typography/color tables don't already cover. Locked decisions from `01-CONTEXT.md` are referenced (`D-NN`), not re-stated.

### Single-page long-scroll structure (D-01)

```
<body class="bg-[--color-bg] text-[--color-fg] font-mono">
  <a class="sr-only focus:not-sr-only ...">Skip to content</a>
  <header role="banner" class="sticky top-0 z-10 ...">     <!-- sticky nav, D-04 -->
  <main id="main" class="mx-auto max-w-[72ch] px-4 md:px-6 py-12">
    <section id="hero">     <!-- whoami block, D-02/D-03 -->
    <section id="about">    <!-- $ cat about.md -->
    <section id="skills">   <!-- $ ls skills/ -->
    <section id="projects"> <!-- $ ls projects/ -->
    <section id="certs">    <!-- $ ls certs/ -->
    <section id="writeups"> <!-- $ ls writeups/ — placeholder per D-11 -->
    <section id="contact">  <!-- $ cat contact.md -->
  </main>
  <footer role="contentinfo" class="mt-16 pb-8 text-[--color-muted]">
</body>
```

- `<main>` carries the 72ch reading width; sticky `<header>` spans full viewport width.
- Each `<section>` has `scroll-margin-top: 80px` so deep-link `#anchor` scrolls land below the sticky nav, not under it.

### Sticky nav layout (D-04 / D-05)

- Single horizontal flex row on `md+` (`>= 768px`): `flex flex-wrap items-center gap-x-4 gap-y-2 px-6 py-3` — all six destinations on one line on typical desktop widths; wraps to a second line if the viewport is narrow enough.
- On `< 768px`: SAME container; flex-wrap naturally pushes overflow to subsequent lines (D-05 — wraps to multiple lines, no horizontal scroll, no hamburger). The literal terminal wraps too.
- Background: `#0d1117` (same as page). Bottom border: `1px solid color-mix(in srgb, var(--color-accent) 20%, transparent)` — a faint accent rule that separates the nav from page content without drawing the eye to it. (This is the ONE permitted opacity-derived accent variant in Phase 1.)
- Each `[bracket]` link: minimum 44×44 hit area via `py-2 px-3 -mx-1` (negative margin keeps the visual gap tight while expanding the click target).

### Anchor-link styling (the seven `[bracket]` link sites)

The same visual + interaction contract applies everywhere a `[bracket]` link appears (sticky nav, hero contact strip, skill tags, project list bullets, 404 sitemap):

| State | Visual |
|-------|--------|
| Default | Bracket glyph and inner label both `#7ee787`. No underline. |
| Hover (mouse) | Inner label gains underline (`text-decoration: underline; text-decoration-color: #7ee787; text-underline-offset: 4px;`). Bracket glyphs stay solid. **No color change.** No background fill. Cursor: `pointer`. |
| Focus-visible (keyboard) | Same as hover, PLUS the focus ring (`outline: 2px solid #79c0ff; outline-offset: 2px;`). |
| Active (clicking) | No transform, no shadow shift. Label briefly inverts: `background-color: var(--color-accent); color: var(--color-bg);` for the duration of the click — a single-frame "selected" flash, the only permitted background-fill-on-link in Phase 1. |
| `aria-current="page"` (current section while scrolling — Phase 1 may skip this; if implemented, no JS-state required, can use `:target` CSS or pure-CSS scroll-snap-active variants) | Bracket glyphs become `▸` `◂` (e.g. `▸about◂`) — a visual cue that doesn't require a color change. Optional in Phase 1; if skipped, document as "deferred to Phase 2 view-toggle work". |

### Skills tag list (D-10)

- Inline `<ul>` of `<li>` items, each containing a single text node like `[python]`, `[bash]`, `[wireshark]`.
- Display: `flex flex-wrap gap-x-3 gap-y-2` — wraps naturally; no scroll bar.
- Each tag is **non-interactive** in Phase 1 (no link, no hover state) — they're statements of fact, not navigation. If Phase 3 adds "click a skill to filter projects", that's a Phase 3 concern.
- **Forbidden:** skill bars, percentages, star ratings, color-coded skill levels (REQUIREMENTS.md "Out of Scope" + research/PITFALLS.md Pitfall 6).

### Project rows (D-13)

Each project is a single line: `<project-slug> — <one-line tagline>`. The `<project-slug>` part is an `<a>` linking to the GitHub repo (or to an in-page `#<slug>` anchor if a future detail block exists — Phase 3). Render order:

```
$ ls projects/
home-lab           — Splunk + pfSense + Wireshark home SOC, write-up in progress
phishing-triage    — Python CLI for parsing Outlook-saved .eml files
└── (more rows here, padded to align with em-dash via tabular-nums monospace)
```

- Use a CSS Grid or simple `<dl>` so the `—` separator visually aligns across rows. (Tabular alignment is free in monospace fonts.)
- Hover state (mouse / focus-visible): row gains `bg-[--color-surface-1]` AND a 2px left border in accent (`border-l-2 border-l-[--color-accent] -ml-2 pl-2`). The combined affordance handles both color-blind users and the "subtle is fine but not invisible" line.
- The honest in-progress project per D-13 uses the **warn**-color leading bracket: `[ ] home-lab — …`.

### Certifications (D-10)

Three render variants based on status. Status enum: `'earned' | 'in-progress' | 'planned'`. Same `<dl>`-style alignment as projects.

```
$ ls certs/
[✓] CompTIA Security+        — earned May 2025
[ ] BTL1 (Blue Team Level 1) — in progress, target Jul 2026
[ ] OSCP                     — planned
```

- The leading bracket `[✓]` is accent green; `[ ]` for in-progress / planned is **warn** amber. The label after is body color. The date is muted color.
- All three variants share one row component; status drives leading-bracket color + trailing copy (per the Copywriting Contract table above).

### Email obfuscation (TXT-04, hero `[email …]` button)

- Pre-click button label: literal string `[email (click to reveal)]` — accent green, same visual as the other three hero brackets so it doesn't read as a "weird one out."
- On click: `obfuscate.ts:revealEmail()` is called. Implementation freedom is delegated to the executor (Claude's-discretion in CONTEXT) — pick a simple reversible encoding (rot13 / base64 / char-array reassembly), document the choice as a code comment.
- After reveal: replaces button with `<a href="mailto:...">[<decoded-address>]</a>`, also writes the address to clipboard via `navigator.clipboard.writeText()`. A small `(copied)` muted-color toast appears for 1.5s next to the link.
- **Fallback if JS disabled:** the button label remains visible but inert. No raw `mailto:` ever appears in the source HTML (TXT-04 / Pitfall 5).

### Skip-to-content link (TXT-05 / Pitfall 10 / Phase-1 success criterion 4)

- First focusable element on the page. Default: visually hidden via `sr-only`; on `:focus`, becomes visible at top-left, accent green text on bg, with focus ring.
- Target: `<main id="main">` so Tab → Enter lands the keyboard user past the sticky nav.

### Focus order (mandatory)

1. Skip-to-content link
2. Sticky nav `[about] [skills] [projects] [certs] [writeups] [contact]` in order
3. Hero `[CV] [GitHub] [LinkedIn] [email …]` in that left-to-right order
4. In-page links by document order
5. Footer `[GitHub]` link

No `tabindex` overrides above 0. `tabindex="-1"` only on `<main>` so the skip link can target it.

### Deep-link behaviour (D-06)

- `?focus=projects` (or any `#projects` direct link) → `scrollIntoView({ behavior: 'smooth', block: 'start' })` to the section's `<h2>`.
- If `prefers-reduced-motion: reduce` → instant jump (`behavior: 'auto'`).
- After scroll, the heading `<h2>` receives `:focus-visible` ring for keyboard users so they know where they landed. Use `<h2 tabindex="-1">` and call `.focus({ preventScroll: true })`.

### Animation contract (TXT-05)

The full inventory of motion in Phase 1. Anything not listed here is forbidden in Phase 1.

| Animation | Element | Trigger | Duration | Reduced-motion behaviour |
|-----------|---------|---------|----------|--------------------------|
| Cursor blink | The `█` glyph at end of hero `$ whoami` line | Always (page load) | 1.0s cycle (0.5s on / 0.5s off) | **Disabled** — render as static solid `█` block when `prefers-reduced-motion: reduce` is matched. |
| Smooth scroll on anchor click | `<html>` scroll | User clicks any `[bracket]` nav link or `[home]` 404 link | ~400ms native | Instant jump. |
| Underline reveal on link hover | `[bracket]` links | Mouse hover / focus-visible | 120ms | Disabled — underline appears instantly. |
| Email "(copied)" toast | Hero email link area | After successful clipboard write | 1.5s visible, 200ms fade | Toast still appears but no fade — appears for 1.5s then disappears instantly. |
| Section heading focus ring on deep-link land | `<h2>` of the targeted section | After `?focus=…` scroll completes | 200ms ring fade-in | Disabled — ring appears instantly. |

**No other animations.** No type-out on body content (D-07). No blinking anywhere except the hero cursor block. No fade-in-on-scroll. No parallax. No carousels. No floating particles. No Matrix rain (REQUIREMENTS.md "Out of Scope").

### Mobile breakpoints

Phase 1 is text + monospace + a long single column — extremely tolerant of viewport width. Two breakpoints only:

| Range | Behaviour |
|-------|-----------|
| `< 768px` (Tailwind `default`) | `<main>` padding `px-4 py-8`. Sticky nav wraps to 2-3 lines naturally (D-05). Body type stays at 16px (do **not** scale down — recruiters need to read this on phones). |
| `≥ 768px` (Tailwind `md`) | `<main>` padding `px-6 py-12`. Sticky nav fits one line on most desktop widths; wraps if necessary. |

`max-w-[72ch]` on `<main>` is in `ch` units, so it self-scales with the font; no separate breakpoint for "tablet" needed.

### Responsive type behaviour

**No responsive font-size step in Phase 1.** Body stays 16px on every breakpoint. Heading stays 20px on every breakpoint. The reasoning: monospace text at 16px on a phone is exactly recruiter-readable, and stepping body up on desktop would create awkward line-length / wrap-rhythm shifts in a long-scroll page. Phase 3 / 4 may revisit if MDX prose pushes for it.
